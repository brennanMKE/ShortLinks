## Description

Implement account recovery for users who have lost access to all their registered passkeys. The flow sends a recovery email with a time-limited link that lets the user register a new credential on their existing account.

## Acceptance Criteria

- [ ] `POST /auth/recover`: accepts `{email}`; looks up user (returns `200` even if email not found, to prevent enumeration); sends recovery email with single-use token (15-min TTL); stores token in `pending_registrations` (reuse same table, distinguish by a `purpose` field or separate token type); returns `200 {"message":"If that email is registered, a recovery link has been sent"}`
- [ ] `GET /auth/recover/verify?token=...`: validates token; returns WebAuthn `PublicKeyCredentialCreationOptions` for adding a new credential (same params as registration: `residentKey: "required"`, `userVerification: "required"`, `authenticatorAttachment` omitted); stores challenge in `webauthn_challenges`
- [ ] `POST /auth/recover/finish`: verifies challenge; inserts new `passkey_credentials` row linked to existing `users.id`; does NOT create a new user; does NOT remove old credentials; creates session; sets cookie
- [ ] Writes `account.recovery_started` audit entry (actor_id NULL) on start, `account.recovered` on finish
- [ ] Writes `credential.added` audit entry on finish
- [ ] `go test ./internal/handlers/` covers: email not found (no enumeration), valid recovery, expired token, invalid challenge

## Implementation

Added a `RecoveryService` (`internal/auth/recovery.go`) and three handler methods
(`RecoverStart`, `RecoverVerify`, `RecoverFinish` in `internal/handlers/auth.go`)
wired at `POST /auth/recover`, `GET /auth/recover/verify`, `POST /auth/recover/finish`
in `cmd/shortlinks/main.go`. The service reuses the #0015 registration plumbing:
the same `registrationOptions()` (residentKey/userVerification required, ES256+RS256,
`authenticatorAttachment` omitted), `NewRegistrationUser`, `webauthn.FinishRegistration`,
`InsertCredential`, `CreateSession`, and `NewSessionToken`/`SetSessionCookie`.

- **`POST /auth/recover`** `{email}` → always returns the same generic
  `200 {"message":"If that email is registered, a recovery link has been sent"}`.
  A single-use recovery token (15-minute TTL) is created and emailed via
  `Mailer.SendRecovery` only when the lowercased email maps to an **active** user.
  Unknown / inactive / malformed emails are swallowed (service returns nil) so the
  response never leaks account existence; only a genuine token-creation or
  mail-delivery failure surfaces as a 500.
- **Storage:** the recovery token reuses the existing `pending_registrations`
  table (email + token + expires_at) — **no migration**. The flow is disambiguated
  by the endpoint and by the WebAuthn challenge `purpose` (`recovery`), not by the
  token row. A new TTL constant `recoveryTTL = 15 * time.Minute` lives in the store.
- **`GET /auth/recover/verify?token=...`** validates the token + 15-min TTL,
  re-confirms the email still maps to an active user, issues a registration
  challenge bound to the **existing** `user_id` (persisted with `purpose='recovery'`,
  linked to the token), and returns the creation options JSON. Unknown/expired
  token or vanished/deactivated account → `400 token invalid or expired`.
- **`POST /auth/recover/finish?token=...`** consumes the recovery challenge
  (recovering the bound `user_id`), runs `FinishRegistration`, **adds** the new
  credential to the existing account (old credentials untouched, no new `users`
  row), deletes the token + challenge, creates a session, sets the cookie, and
  returns `200 {"user_id":...}`.

Store gained `CreateRecoveryToken`, `LookupRecoveryToken`, `SaveRecoveryChallenge`,
and `ConsumeRecoveryChallenge` rather than duplicating data access. `NewAuthHandler`
now takes a third `recoverer` dependency.

Audit entries `account.recovery_started`, `account.recovered`, and
`credential.added` are left as `TODO(#0025)` no-ops, matching the rest of the
auth ceremonies.

## Gotchas

- WebAuthn registration verification only requires the session handle and the
  user handle to be internally consistent (both come from a freshly built
  `RegistrationUser`); it does not compare against any previously issued handle.
  So, exactly as registration finish does, recovery finish can build a brand-new
  `RegistrationUser` for the email — the new credential simply attaches to the
  existing `user_id`. `passkey_credentials` stores no user handle, so nothing ties
  the new credential to the account's original login handle; the `user_id` FK is
  the only link, which is what keeps "add to existing account" correct.
- The `pending_registrations` row carries its own `expires_at`, so the 15-min TTL
  is enforced by the value written at creation, not by the lookup constant — but a
  distinct `recoveryTTL` is still used for both the token and the challenge so the
  recovery path never depends on the 5-min registration TTL.
