## Description

Implement the admin routes for user management: listing all users, viewing user detail, deactivating a user (with reason and note), and reactivating a user. All routes require admin auth.

## Acceptance Criteria

- [ ] `GET /admin/users`: returns paginated list with `{id, email, is_admin, active, created_at, last_login_at}`; requires admin auth
- [ ] `GET /admin/users/{id}`: returns user detail plus their link count and passkey count; requires admin auth
- [ ] `POST /admin/users/{id}/deactivate`: body `{reason, note?}`; validates `reason` is one of the six defined values (malware_distribution, phishing, spam, harassment, terms_violation, other); `note` required when `reason = "other"`; sets `users.active = false`; deletes all active sessions for that user; writes `account.deactivated` audit entry with `{reason, note}` metadata; returns `200`; returns `400` if trying to deactivate an admin user
- [ ] `POST /admin/users/{id}/reactivate`: body `{note?}`; sets `users.active = true`; writes `account.reactivated` audit entry; returns `200`
- [ ] All routes require admin auth (`RequireAdmin` middleware from #0017)
- [ ] `go test ./internal/handlers/` covers: list users, deactivate with valid reason, deactivate requires note for "other", cannot deactivate admin, reactivate

## Implementation

Added the admin user-management Store queries and HTTP handler, all behind
`RequireSession` + `RequireAdmin`.

**Store (`internal/auth/users.go`)** — new methods on `*auth.Store`:

- `ListUsers` — every account newest-first as `ManagedUser{id, email, is_admin, active, created_at, last_login_at}`.
- `GetUser` — one account as `UserDetail` (the row plus correlated link/passkey counts); `ErrUserNotFound` → 404.
- `DeactivateUser` — opens a transaction, locks the row `FOR UPDATE`, refuses admins (`ErrUserIsAdmin`) and already-inactive accounts (`ErrUserAlreadyInactive`), sets `active=false`, `DELETE`s **all** of the user's sessions, then writes the `account.deactivated` audit via the auditor's `WriteTx` so the row commits atomically with the change. Links are never touched.
- `ReactivateUser` — transactional set `active=true` + `account.reactivated` `WriteTx`; `ErrUserAlreadyActive` for a no-op; no session restoration.

New sentinel errors: `ErrUserIsAdmin`, `ErrUserAlreadyInactive`, `ErrUserAlreadyActive` (reuses existing `ErrUserNotFound`).

**Handler (`internal/handlers/users.go`)** — `AdminUsersHandler` with `List`, `Get`, `Deactivate`, `Reactivate`. Deactivate validates `reason` against the six PRD values (400 otherwise), requires a non-empty `note` when `reason=="other"` (400 otherwise), refuses self-deactivation (403), and maps the store's admin-target/conflict errors to 403/409. It builds the `account.deactivated` `{reason, note}` / `account.reactivated` `{note}` audit entry (actor = acting admin from context, user/target = the target id, `target_type=user`, ip via `clientIP`) and passes it through to the store so the write is in-band. This wires the `ActionAccountDeactivated`/`ActionAccountReactivated` constants left without a call site by #0025.

**Routes (`cmd/shortlinks/main.go`)** — `GET /admin/users`, `GET /admin/users/{id}`, `POST /admin/users/{id}/deactivate`, `POST /admin/users/{id}/reactivate`, all wrapped with `requireAdmin`.

## Gotchas

- The audit row is written via `audit.Logger.WriteTx` *inside the store's
  transaction*, not fire-and-forget `Record`, because the PRD requires the
  deactivation and its audit to be atomic. The handler builds the `audit.Entry`
  (it owns the request context for actor id + IP) and passes it down to the store,
  which supplies the open `pgx.Tx`.
- Deactivation is intentionally **not** idempotent: a second deactivate of an
  already-inactive user returns 409 rather than no-op, so the audit log never
  accrues a misleading duplicate `account.deactivated`. Reactivate mirrors this
  (409 `ErrUserAlreadyActive`).
- Self-deactivation is blocked at the handler (403) before the store call; since
  the acting principal is always an admin, the store's admin guard would catch it
  too, but the explicit check gives a clearer error.
