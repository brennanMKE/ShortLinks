## Description

Implement per-user URL deduplication in the link creation path. When a user submits a destination URL they have previously shortened (and not had denied), return the existing link rather than inserting a duplicate. Inactive links are reactivated.

## Acceptance Criteria

- [x] Dedup query runs after filter check and only when no custom key was supplied:
  ```sql
  SELECT * FROM links WHERE user_id = $1 AND destination_url = $2 AND denied_reason = 0 LIMIT 1
  ```
- [x] Active link found → return existing link with `"duplicate": true`; no insert; no SSE event (SSE seam `broadcast=false`; audit seam #0025 documented to write `link.created` with `{duplicate:true}`)
- [x] Inactive link found (`active = false`, `denied_reason = 0`) → set `active = true`; return reactivated link with `"duplicate": true`; broadcast `link.created` SSE (seam `broadcast=true`; audit seam documented as `link.reactivated`)
- [x] No match → insert new link; return with `"duplicate": false`; broadcast `link.created` SSE (seam `broadcast=true`)
- [x] Custom aliases (`key` provided in request body) bypass dedup entirely — insert is attempted regardless
- [x] Tests cover: active duplicate, inactive reactivation, no match, custom alias bypasses dedup (plus per-user isolation)

## Implementation

Per-user URL deduplication was added to the generated-key `POST /api/links`
path (the seam left by #0022). Custom aliases are unchanged — they bypass
dedup entirely and still go through `CreateLink` (a key clash still returns
409).

The dedup decision and any write happen atomically inside one transaction so
two concurrent creates of the same URL cannot both insert a duplicate:

- New method `links.Store.CreateOrReactivateLink(ctx, NewLink, genKey)`:
  - Begins a tx, then row-locks the existing non-denied link for
    `(user_id, destination_url)` with `... AND denied_reason = 0 LIMIT 1 FOR UPDATE`.
  - **Active match** → return it unchanged, `OutcomeActiveDuplicate` (no write).
  - **Inactive match** → `UPDATE links SET active = TRUE`, `OutcomeReactivated`.
  - **No match** → mint a unique key (collision check runs inside the same tx),
    insert a fresh active link, `OutcomeInserted`.
  - Returns a new `links.CreateOutcome` enum so the handler can set `duplicate`
    and decide whether to fire the post-create seams.
- Handler `Create` maps the outcome: `duplicate=true` for active-duplicate and
  reactivated, `false` for a fresh insert. The response shape is identical in
  all cases (a full link object). A `broadcast` boolean is wired for the #0026
  SSE seam: true on insert or reactivation, false on a pure active duplicate
  (per the PRD).

Seam state afterward: **#0024** (filter, runs before dedup) and **#0025**
(audit) are left as documented TODO seams, unchanged in ordering. **#0026**
(SSE) is still a TODO but now has its `broadcast` predicate wired and the seam
comments updated to reflect insert/reactivation-vs-active-duplicate.

## Gotchas

- Go method-set matching is structural and uses the literal parameter type:
  a *named* `genKeyFn` type on the `Store` method did not satisfy the handler
  interface's inline `func(...)` parameter, so the method signature uses the
  literal function type (matching `links.GenerateUniqueKey` exactly).
- Key generation runs *inside* the dedup transaction (its `EXISTS` check uses
  the tx) so the collision check and insert see a consistent snapshot; the
  no-match branch is the only one that calls `genKey`.
