## Description

Add per-IP rate limiting to the authentication endpoints most vulnerable to abuse. Uses golang.org/x/time/rate with an in-memory map of per-IP limiters.

## Acceptance Criteria

- [ ] `internal/middleware/ratelimit.go` implements `RateLimit(r rate.Limit, b int)` middleware factory that returns an `http.Handler` wrapper
- [ ] Extracts client IP from `X-Forwarded-For` header (set by Apache proxy) with fallback to `RemoteAddr`
- [ ] Returns `429 Too Many Requests` with `{"error":"rate_limit_exceeded"}` when limit is exceeded
- [ ] Applied to the following routes with the specified limits:
  - `POST /auth/register/start`: 3 requests / hour / IP
  - `POST /auth/login/finish`: 10 requests / minute / IP
  - `POST /auth/recover`: 3 requests / hour / IP
- [ ] Per-IP limiters are stored in a `sync.Map` (or mutex-protected map); stale entries cleaned up periodically (at minimum on access, or via a background ticker)
- [ ] `go test ./internal/middleware/` covers: under limit passes, over limit returns 429

## Implementation

`internal/middleware/ratelimit.go` adds a `RateLimiter` backed by
`golang.org/x/time/rate`. `NewRateLimiter(r rate.Limit, burst int)` holds a
mutex-guarded `map[string]*rate.Limiter`, creating a per-IP limiter lazily on
first sight (`limiterFor`). The `Middleware(next http.Handler) http.Handler`
method extracts the client IP, looks up/creates that IP's limiter, and on
`Allow() == false` returns `429 Too Many Requests` with the JSON body
`{"error":"rate_limit_exceeded"}` (reusing the package's existing
`writeJSONError`); otherwise it calls `next`.

IP extraction reuses the redirect handler's approach (#0009): first hop of
`X-Forwarded-For` when present (the real client behind Apache), else
`RemoteAddr` with the port stripped — so a client is bucketed consistently
across the service.

`cmd/shortlinks/main.go` constructs and attaches three limiters matching the
PRD Phase 2 rates, with burst equal to the per-window allowance so a fresh IP
gets its full quota immediately:

- `POST /auth/register/start` — `rate.Every(time.Hour/3)`, burst 3 (3/hour/IP)
- `GET /auth/login/start` — `rate.Every(time.Minute/10)`, burst 10 (10/min/IP)
- `POST /auth/recover` — `rate.Every(time.Hour/3)`, burst 3 (3/hour/IP)

Note: the PRD lists `login/start` (not `login/finish`, as the acceptance
criteria here did) as the rate-limited login endpoint, and the route is
actually registered as `GET /auth/login/start`. The limiter is attached to the
real registered route per the PRD, which is authoritative for the rates.

## Gotchas

- Per-IP limiters are never evicted — the map grows by one entry per distinct
  client IP for the lifetime of the process. For this single-instance service
  behind Apache the distinct-IP set is bounded enough that the simplicity wins,
  so no eviction/TTL was added (the issue explicitly allows a simple map). If
  this ever fronts untrusted high-cardinality traffic, add a background sweep or
  an LRU bound.
- The acceptance criteria named `POST /auth/login/finish`, but the PRD (the
  authoritative source for rates) names `login/start`, and only
  `GET /auth/login/start` is registered. The limiter was attached there; the
  `login/finish` handler is left unthrottled.
