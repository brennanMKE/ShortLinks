## Description

The integration tests in `internal/auth`, `internal/handlers`, `internal/middleware`,
`internal/db`, and `cmd/shortlinks` all connect to the same `TEST_DATABASE_URL`
(`shortlinks_test`) and `TRUNCATE` the shared tables (`users`, `passkey_credentials`,
`sessions`, etc.) in their setup helpers. `go test` runs package test binaries concurrently by
default, so while one package has seeded rows and is mid-assertion, another package's `TRUNCATE`
wipes them — producing foreign-key violations and false failures.

`go test ./...` (the verify command documented in `issues/Issues.md`) is therefore unreliable.
The only current workaround is `go test -p 1 ./...`, which serializes packages.

This was discovered while resolving #0019.

## Steps to reproduce

1. Ensure `shortlinks_test` is migrated and `TEST_DATABASE_URL` is exported.
2. Run `go test ./...` (no `-p 1`).
3. Observe intermittent failures such as `passkey_credentials rows = 0, want 1`,
   `insert or update on table "passkey_credentials" violates foreign key constraint`, and
   `first user should be promoted to admin`.

## Expected

`go test ./...` passes reliably without requiring `-p 1`.

## Actual

Concurrent package execution against one shared mutable database causes cross-package data
corruption and non-deterministic failures.

## Acceptance Criteria

- [x] Integration test packages no longer interfere when run concurrently.
- [x] `go test ./...` (no `-p 1`) passes reliably across repeated runs.
- [x] The fix does not require every developer to remember a special flag.

## Notes

Likely fix: serialize DB access across packages with a PostgreSQL session-level advisory lock
acquired in a shared test helper invoked from each package's `TestMain`, so only one
DB-touching package runs at a time even under `go test`'s default concurrency. Affected
packages: `internal/auth`, `internal/handlers`, `internal/middleware`, `internal/db`,
`cmd/shortlinks`. None currently define `TestMain`.

## Root cause

`go test ./...` compiles each package into its own test binary and runs those binaries
concurrently (up to `-p`, default GOMAXPROCS). Five packages connect to the single shared
`shortlinks_test` database and `TRUNCATE` the same tables in their per-test setup. With two
package binaries running at once, package B's `TRUNCATE users … CASCADE` deletes the rows
package A just seeded and is mid-assertion on — yielding foreign-key violations and bogus
assertion failures. The data layer was correct; only the *test harness* assumed exclusive DB
access it didn't have.

## Fix

Added `internal/testdb` with a single `Lock()` function that opens a dedicated connection and
takes a PostgreSQL **session-level advisory lock** (`pg_advisory_lock`) on a fixed key, returning
a release function. Each of the five DB-touching packages gained a `main_test.go` with a
`TestMain` that calls `Lock()`, runs `m.Run()`, then releases. Because the lock is held for the
whole package run and every package uses the same key, the package test binaries serialize
against the shared DB even though `go test` still launches them concurrently. When
`TEST_DATABASE_URL` is unset, `Lock()` is a no-op and the DB tests skip themselves, so non-DB
runs keep full parallelism. No production code changed; no special flag required.

## Verification

`go build ./...` and `go vet ./...` clean. With `TEST_DATABASE_URL` pointed at `shortlinks_test`,
`go test -count=1 ./...` (default concurrency, **no `-p 1`**) was run three consecutive times —
all eight packages reported `ok` every time, with zero FK violations or assertion failures. The
same command reliably FAILED before the fix (see Steps to reproduce).

## Files changed

- `internal/testdb/testdb.go` — new helper; `Lock()` acquires/releases the shared advisory lock (no-op without `TEST_DATABASE_URL`).
- `internal/auth/main_test.go` — `TestMain` serializing the package via `testdb.Lock()`.
- `internal/handlers/main_test.go` — same.
- `internal/middleware/main_test.go` — same.
- `internal/db/main_test.go` — same.
- `cmd/shortlinks/main_test.go` — same.

## Gotchas

- The lock is **session-level**, so it must be held on a long-lived `*pgx.Conn` (not a pooled
  connection that may be returned between statements) — `testdb` keeps its own dedicated conn open
  until release.
- Any *future* package that mutates the shared test DB must also add the `TestMain` + `testdb.Lock()`
  shim, or it will reintroduce the race. Consider this when adding DB-backed tests in new packages.
- This serializes DB-backed packages (effectively `-p 1` for them) but leaves non-DB packages
  (`cache`, `config`, `links`) running in parallel, so the suite is correct without being fully
  serialized.
