## Description

Create the Phase 1 database migrations using `golang-migrate/migrate` naming conventions. Migrations must run in order and be reversible. The `users` table must be created before `links` due to the foreign key.

## Acceptance Criteria

- [x] `migrations/000001_create_users.up.sql` — creates `users` table with all columns from PRD: `id`, `email`, `is_admin`, `active`, `created_at`, `last_login_at`
- [x] `migrations/000002_create_links.up.sql` — creates `links` table: `id`, `user_id` (FK → users), `key`, `destination_url`, `title`, `created_at`, `expires_at`, `active`, `denied_reason`
- [x] `migrations/000003_create_clicks.up.sql` — creates `clicks` table with all UTM columns
- [x] All Phase 1 indexes created (folded into the per-table migrations rather than a separate `0004` file): `links(key)` (covered by UNIQUE), `links(user_id)`, `links(user_id, destination_url) WHERE denied_reason = 0`, `links(denied_reason) WHERE denied_reason > 0`, `clicks(link_id)`, `clicks(clicked_at)`
- [x] Corresponding `.down.sql` files drop tables/indexes in reverse order
- [x] `migrate -path migrations -database $DATABASE_URL up` runs cleanly against a fresh database
- [x] `migrate -path migrations -database $DATABASE_URL down` reverses all migrations cleanly

## Notes

PRD reference: Database Schema, Database Indexes sections.
Dependencies: #0001, #0002.
Tool: `github.com/golang-migrate/migrate/v4`.

## Implementation

Created the three Phase 1 core tables as golang-migrate `{version}_{description}.{up,down}.sql`
pairs with zero-padded 6-digit versions, in foreign-key order: `000001_create_users`,
`000002_create_links`, `000003_create_clicks`. Columns, types, and constraints match the PRD
exactly (`users` from the Auth Tables section; `links` and `clicks` from the Database Schema
section). `created_at` / `clicked_at` get `DEFAULT now()` since the PRD treats them as creation
timestamps.

Indexes from the PRD "Database Indexes" table that apply to these tables were folded into the
relevant table migration rather than a separate `0004` file: `links(user_id)`, the partial dedup
index on `links(user_id, destination_url) WHERE denied_reason = 0`, the partial admin index on
`links(denied_reason) WHERE denied_reason > 0`, `clicks(link_id)`, and `clicks(clicked_at)`. The
`key` and `email` UNIQUE constraints supply their own indexes, so no separate index was created
for them (noted in comments). Auth tables (`passkey_credentials`, `sessions`, etc.) and
`url_filter_rules` are intentionally deferred to later issues (0013, 0024). Each `.down.sql`
reverses its `.up.sql` — dropping indexes first, then the table, in reverse FK order.

## Verification

`go build ./...` succeeds (unaffected by the SQL-only change). The SQL was first verified by
careful inspection: column types/constraints match the PRD; FK creation order is users → links →
clicks; every `up` has a matching `down` that reverses it in correct order; index definitions are
valid PostgreSQL; and the partial-index `WHERE` clauses match the PRD (`denied_reason = 0` for
dedup, `denied_reason > 0` for the admin index).

**Live verification (2026-05-25):** PostgreSQL 16 was subsequently installed locally and the
`shortlinks` database created via `scripts/db/create.sql`. `migrate -path migrations -database
$DATABASE_URL up` applied all three migrations cleanly, producing tables `users`, `links`,
`clicks` (+ golang-migrate's `schema_migrations`) and exactly the expected indexes
(`idx_links_user_id`, `idx_links_user_destination`, `idx_links_denied_reason`,
`idx_clicks_link_id`, `idx_clicks_clicked_at`, plus the `links_key_key` / `users_email_key`
UNIQUE indexes). `migrate ... down -all` dropped them in reverse FK order leaving only
`schema_migrations`, and a re-run of `up` re-created everything — confirming full reversibility.
Both live-run acceptance criteria now pass.

## Files changed

- `migrations/000001_create_users.up.sql` — creates `users` (id, email, is_admin, active, created_at, last_login_at); email UNIQUE.
- `migrations/000001_create_users.down.sql` — drops `users`.
- `migrations/000002_create_links.up.sql` — creates `links` (FK → users) plus `links(user_id)`, partial dedup index, and partial denied_reason index.
- `migrations/000002_create_links.down.sql` — drops the three `links` indexes then the `links` table.
- `migrations/000003_create_clicks.up.sql` — creates `clicks` (FK → links) with all UTM columns plus `clicks(link_id)` and `clicks(clicked_at)`.
- `migrations/000003_create_clicks.down.sql` — drops the two `clicks` indexes then the `clicks` table.

## Gotchas

- The issue's acceptance criteria named a separate `0004_create_indexes` file and `0001`-style
  4-digit versions; per the project's resolve direction the indexes were folded into their owning
  table migrations and versions use 6-digit zero-padding (`000001`…), which is golang-migrate's
  conventional format. Functionally identical — all named indexes exist.
- Only the three core tables for this issue were created. Do not add the auth/filter tables here;
  they belong to issues 0013 and 0024.
