## Description

SES SMTP credentials were added to `/etc/shortlinks/config.env`, but the account's sending status was never verified. A sandboxed SES account can only send to verified recipient addresses and is capped at 200 messages/day; an unverified sending domain causes silent drops. Until this is confirmed, registration/recovery emails may be accepted by the app yet never delivered — independent of the frontend work in #0041–#0043.

## Long Description

Operational checklist, not code. From `docs/email_setup.md` the open items are: domain identity (`sstools.co`) verified, Easy DKIM CNAMEs resolving, custom MAIL FROM (`mail.sstools.co`) MX+SPF resolving, DMARC TXT present, and production access granted (or `admin@example.com` verified as a sandbox recipient). Note the app reports a successful SMTP handoff as success — so a sandbox rejection at send time would surface as a 500, while a post-acceptance drop/bounce would not surface in the app at all.

## How to verify (runnable — region us-east-1)

Run on the EC2 box (or any host with AWS creds). Region derived from `SES_SMTP_HOST=email-smtp.us-east-1.amazonaws.com`.

```bash
# 1. Sandbox vs production
aws sesv2 get-account --region us-east-1 \
  --query '{production:ProductionAccessEnabled, sendingEnabled:SendingEnabled}'

# 2. Domain identity verified + DKIM
aws sesv2 get-email-identity --email-identity sstools.co --region us-east-1 \
  --query '{verified:VerifiedForSendingStatus, dkim:DkimAttributes.Status, mailFrom:MailFromAttributes.MailFromDomainStatus}'

# 3. Is the admin recipient suppressed (from a prior bounce)?
aws sesv2 get-suppressed-destination --email-address "$ADMIN_EMAIL" --region us-east-1 \
  2>&1 || echo "not suppressed (good)"

# 4. DNS resolves
dig +short TXT _dmarc.sstools.co
dig +short MX  mail.sstools.co
dig +short TXT mail.sstools.co

# 5. If still in sandbox and not yet granted production: verify the recipient so test sends work
aws sesv2 create-email-identity --email-identity "$ADMIN_EMAIL" --region us-east-1
#   ...then click the verification link SES emails to $ADMIN_EMAIL

# 6. Real send test (after identity/recipient verified)
aws sesv2 send-email --region us-east-1 \
  --from-email-address "ShortLinks <noreply@sstools.co>" \
  --destination ToAddresses="$ADMIN_EMAIL" \
  --content 'Simple={Subject={Data="SES test"},Body={Text={Data="delivery works"}}}'
```

Pass criteria: domain `VerifiedForSendingStatus=true`, DKIM `SUCCESS`, recipient not suppressed, and the step-6 message lands in the inbox with `dkim=pass`/`spf=pass`/`dmarc=pass`. Then the **Recover account** flow's email (#0044) will actually arrive.
