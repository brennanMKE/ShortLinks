## Description

Implement a `Mailer` interface and AWS SES SMTP implementation used to send transactional emails: magic-link verification emails and account recovery emails.

## Acceptance Criteria

- [ ] `internal/auth/mailer.go` defines a `Mailer` interface with at minimum: `SendVerification(ctx, toEmail, token string) error` and `SendRecovery(ctx, toEmail, token string) error`
- [ ] `internal/auth/ses_mailer.go` implements `Mailer` using SMTP (host/port/username/password from `Config`)
- [ ] Verification email body includes the full magic link: `{BASE_URL}/auth/register/verify?token={token}`
- [ ] Recovery email body includes the full recovery link: `{BASE_URL}/auth/recover/verify?token={token}`
- [ ] Uses `net/smtp` with STARTTLS on port 587 (SES_SMTP_PORT from config)
- [ ] `From` header uses `EMAIL_FROM` from config
- [ ] A `NoOpMailer` stub is provided for testing (logs to stdout, does not send)
- [ ] `go test ./internal/auth/` covers mailer with `NoOpMailer` (no real SES call required)

## Notes

Dependencies: #0005 (config loader), #0013 (auth tables — registration flow calls mailer).

## Implementation

- `internal/auth/mailer.go` defines the `Mailer` interface with
  `SendVerification(ctx, toEmail, token)` and `SendRecovery(ctx, toEmail, token)`,
  plus a `NoOpMailer` stub that logs the rendered link to stdout and never sends.
  The link builders produce `{BASE_URL}/auth/register/verify?token={token}` and
  `{BASE_URL}/auth/recover/verify?token={token}`.
- `internal/auth/ses_mailer.go` implements `Mailer` as `SESMailer`, constructed
  from `*config.Config` (`SES_SMTP_*`, `EMAIL_FROM`, `BASE_URL`). It composes an
  RFC 5322 `text/plain` message (From/To/Subject/Date/MIME headers + CRLF body)
  and dispatches it through an injectable `sendMailFunc` (same signature as
  `smtp.SendMail`). The production default `starttlsSendMail` dials the SES SMTP
  endpoint, upgrades with STARTTLS, and authenticates with `smtp.PlainAuth`. The
  SMTP envelope sender is the bare address parsed from the display-name `From`.

The transport seam is what makes the mailer unit-testable without SES.

Honest note: there are no real AWS SES credentials and no outbound SMTP server
in this environment, so no real email was sent. Delivery is verified against an
injected recording transport and an in-process fake SMTP listener; the STARTTLS
upgrade is conditional (the fake advertises no extensions) so the real-transport
test exercises dial/MAIL/RCPT/DATA without performing TLS.

## Gotchas

- The `From` header carries the display-name form (`ShortLinks <noreply@sstools.co>`),
  but the SMTP envelope sender (`MAIL FROM`) must be the bare address — `fromAddress`
  parses the angle-bracketed address out of the configured `EMAIL_FROM`.
- No new module dependencies: everything is stdlib (`net/smtp`, `crypto/tls`),
  so `go.mod`/`go.sum` are unchanged.
