## Description

The seed command and the registration flow contradict each other for the admin account. `seed.go:71` inserts a `users` row for `ADMIN_EMAIL` (no passkey). `StartRegistration` then rejects that email because `EmailRegistered` returns true â†’ returns `ErrEmailRegistered`, and the handler shows the generic "Check your email" message **without sending anything** (`handlers/auth.go:96-98`). So the seeded admin can never enroll a passkey via registration, and `FinishRegistration`'s admin-promotion logic (`registration.go:215`) never runs. Recovery is the only backend path to add a passkey to an existing account â€” but it has no UI (#0043). Net effect: the admin account exists but has no way to obtain a passkey through the product.

## Long Description

Observed live: user row `id=1 admin@example.com is_admin=t active=t`, created by the seed on 2026-05-27, with zero passkeys and `last_login_at` null. Every registration attempt for that email returns the green decoy and sends no email â€” which is what made the missing-email symptom look like an SES problem.

Two compounding gaps:
1. **Seed vs. registration ownership.** Either the seed should *not* create the admin user (let the first registration create + auto-promote it, per PRD lines 345 and `registration.go:215`), or the seeded-but-passkey-less account must have a supported enrollment path (recovery UI, #0043), or registration must handle "email exists but has no credential" as an enroll-rather-than-reject case.
2. **No first-admin open.** `registrations_enabled` defaults `false` with no bypass, so even a non-seeded fresh install cannot register its first admin without a manual DB flip. The bootstrap story needs to be defined and documented.

The user is the admin, so this is the special case that must be solved for first login to work at all.

## Expected behavior

## Actual behavior

Seeded admin is locked out: registration short-circuits (no email), recovery has no UI, and the gate defaults closed.
