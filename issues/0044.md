## Description

The seed command and the registration flow contradict each other for the admin account. `seed.go:71` inserts a `users` row for `ADMIN_EMAIL` (no passkey). `StartRegistration` then rejects that email because `EmailRegistered` returns true â†’ returns `ErrEmailRegistered`, and the handler shows the generic "Check your email" message **without sending anything** (`handlers/auth.go:96-98`). ## Long Description

Every registration attempt for that email returns the green decoy and sends no email â€” which is what made the missing-email symptom look like an SES problem.

Two compounding gaps:

The user is the admin, so this is the special case that must be solved for first login to work at all.

## Root cause

`seed` creates the admin `users` row (no passkey); `StartRegistration` rejects an already-registered email (`registration.go:85-90`) and the handler returns a generic "check your email" without sending (`handlers/auth.go:96-98`), so the admin could never enroll via registration. The fix is choosing the right enrollment path, not removing the conflict: recovery (#0018/#0043) adds a credential to an existing account and is not gated by `registrations_enabled`.

## Fix

No change to the recovery code was needed â€” verified it already handles a zero-credential account and preserves `is_admin`. Added `TestRecoveryEndToEnd_AdminBootstrap_ZeroCredentials` proving startâ†’verifyâ†’finish for a seeded admin with zero passkeys creates a credential, keeps `is_admin=true`, creates a session, and adds no duplicate user. Rewrote DEPLOYMENT.md's "First admin login" section to document the recovery-based bootstrap (recovery is ungated; registration rejects an existing email; `registrations_enabled` only matters for other users; SES must work). Added a one-line hint to `seed` pointing the admin at the Recover account link.
