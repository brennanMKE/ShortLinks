## Description

Add the Phase 2 database migrations for all authentication and audit tables. Runs after the Phase 1 migrations (users, links, clicks) defined in issue #0006.

## Acceptance Criteria

- [ ] Migration file `migrations/0003_auth_tables.up.sql` creates: `passkey_credentials`, `webauthn_challenges`, `pending_registrations`, `sessions`, `settings`, `audit_log`
- [ ] Migration file `migrations/0003_auth_tables.down.sql` drops all six tables in reverse dependency order
- [ ] `settings` table seeded with `INSERT INTO settings (key, value, updated_at) VALUES ('registrations_enabled', 'false', now()) ON CONFLICT DO NOTHING`
- [ ] All columns and constraints match the PRD schema exactly (see Database Schema — Auth Tables)
- [ ] Indexes applied:
  - `sessions(token)` — covered by UNIQUE constraint
  - `passkey_credentials(credential_id)` — covered by UNIQUE constraint
  - `webauthn_challenges(expires_at)`
  - `pending_registrations(expires_at)`
  - `audit_log(user_id, created_at DESC)`
  - `audit_log(created_at DESC)`
  - `audit_log(action)`
  - `url_filter_rules(active)`
- [ ] `url_filter_rules` table created with columns: `id`, `pattern`, `reason_code`, `description`, `active`, `created_by`, `created_at`
- [ ] Migration applies cleanly with `migrate -path migrations -database $DATABASE_URL up`
- [ ] Migration rolls back cleanly with `migrate -path migrations -database $DATABASE_URL down`

## Notes

Dependencies: #0006 (Phase 1 migrations must run first — `users` table must exist).

## Implementation

Continued the existing 6-digit `golang-migrate` numbering (`000001`–`000003`
for users/links/clicks) with four new `{up,down}.sql` pairs covering the six
auth/audit tables. Columns, types, nullability, defaults, and FKs match the
PRD "Database Schema — Auth Tables" and "Database Indexes" sections exactly.

FK ordering is honoured within `000004`: `pending_registrations` is created
first so its `token UNIQUE` constraint exists before `webauthn_challenges`
references it via `pending_registration_token TEXT REFERENCES
pending_registrations(token)`. The matching `down` files drop indexes and
tables in reverse order.

- `000004_create_auth_credentials` — `pending_registrations`,
  `passkey_credentials`, `webauthn_challenges` (+ `expires_at` sweep indexes).
- `000005_create_sessions` — `sessions` (token UNIQUE covers its lookup index).
- `000006_create_settings` — `settings`, seeded with
  `('registrations_enabled','false', now()) ON CONFLICT DO NOTHING`; the down
  removes the seed row and drops the table.
- `000007_create_audit_log` — `audit_log` with indexes
  `(user_id, created_at DESC)`, `(created_at DESC)`, and `(action)`.

Note: this issue's original Acceptance Criteria referenced legacy `0003_`
filenames and a `url_filter_rules` table. Per the authoritative PRD and the
current `migrations/` layout, the six auth/audit tables are delivered here as
`000004`–`000007`; `url_filter_rules` belongs to the filters work and is out of
scope for this migration.

## Gotchas

- `webauthn_challenges.pending_registration_token` requires
  `pending_registrations` (and its `token UNIQUE`) to exist first — both live in
  `000004` in the correct order.
- `audit_log` has two FKs to `users(id)` (`actor_id` and `user_id`), both
  nullable for pre-auth / admin-on-behalf events.
