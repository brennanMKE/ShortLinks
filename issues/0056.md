# 0056 — Umbrella: Local dev runtime to build & run Short Links on macOS (no systemd, no Postgres) with iPhone Simulator testing

| | |
|---|---|
| **Status** | open |
| **Module** | infra / db / auth / web |
| **Platform** | macOS |
| **First seen** | 2026-06-22 |

## Description

Make it possible to build and run the Short Links website directly on this Mac for fast local iteration and **visual verification** — without provisioning PostgreSQL or systemd, and with a mock signed-in account so `localhost` reaches the dashboard. With a local server running, the iOS Simulator's Safari can load it to verify responsive layout and JS / native-control behavior. This umbrella tracks the sub-issues that together deliver that loop.

## Motivation

Several recent UI changes (#0050 responsive, #0051 dark mode, and especially #0053 — the Expires field width/height) could **not** be visually verified in this environment, which forced repeated deploy-and-guess cycles against production. A local build → run → inspect loop, including real mobile Safari via the iOS Simulator, closes that gap and would have caught the #0053 rendering issues immediately.

## Sub-issues

- [ ] [#0057](0057.md) — Storage abstraction + JSON-file dev backend (replace Postgres for dev)
- [ ] [#0058](0058.md) — Dev-only auth bypass with a mock admin user (skip the passkey ceremony on localhost)
- [ ] [#0059](0059.md) — One-command local run (Makefile/script) without systemd or Postgres
- [ ] [#0060](0060.md) — iPhone Simulator (iOS Safari) verification workflow for layout & JS

## Acceptance criteria (umbrella)

- [ ] A single command builds and serves the app on `localhost` with no Postgres, no systemd, no migrations
- [ ] Opening `localhost` lands on the dashboard as a mock admin (no real passkey)
- [ ] App data persists to JSON files (or in-memory) in dev; production stays Postgres-backed and unchanged
- [ ] The iOS Simulator's Safari can load the local URL and verify responsive layout / native-control rendering (e.g. the #0053 datetime field)

## Guardrails

Dev-only: the JSON backend and the auth bypass MUST be impossible to enable in production — guarded by explicit dev config and/or a build tag, and refused when `BASE_URL`/RP id is not localhost. PostgreSQL + WebAuthn passkeys remain the sole production path.
