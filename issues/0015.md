## Description

Implement the three-step passkey registration flow: email submission, magic-link verification, and WebAuthn attestation completion. iCloud Keychain compatible (authenticatorAttachment omitted). First registered email matching ADMIN_EMAIL is promoted to admin.

## Acceptance Criteria

- [x] `POST /auth/register/start`: checks `registrations_enabled` setting; returns `403` if disabled; validates email not already registered; creates `pending_registrations` row (5-min TTL); sends verification email via `Mailer`; returns `200 {"message":"Check your email"}`
- [x] `GET /auth/register/verify?token=...`: validates token exists and not expired; issues WebAuthn `PublicKeyCredentialCreationOptions` with `residentKey: "required"`, `userVerification: "required"`, `authenticatorAttachment` omitted; stores challenge in `webauthn_challenges` linked to `pending_registration_token`; returns options JSON
- [x] `POST /auth/register/finish`: verifies challenge; stores `passkey_credentials` row; creates `users` row with `active=true`; if no other users exist OR email matches `ADMIN_EMAIL`, sets `is_admin=true`; deletes `pending_registrations` row; creates session; sets `HttpOnly; Secure; SameSite=Strict` cookie; returns `200`
- [x] Uses `github.com/go-webauthn/webauthn` library
- [x] `pubKeyCredParams`: ES256 (`-7`) preferred, RS256 (`-257`) accepted
- [x] `user.id` in WebAuthn options is a random 16-byte value (not derived from email)
- [ ] Writes `account.registration_started` audit entry on start, `account.registered` on finish â€” deferred to #0025 (audit write path); marked with TODOs at the call sites
- [ ] Writes `credential.added` audit entry with `{device_name, aaguid}` metadata â€” deferred to #0025; TODO in place (device_name + aaguid are already persisted on the credential row)
- [x] `go test ./internal/auth/` and `go test ./internal/handlers/` cover the happy path and error cases (duplicate email, expired token, invalid challenge)

## Implementation

The ceremony is split into a reusable service layer (`internal/auth`) and a thin
HTTP shell (`internal/handlers`). The service is the auth foundation #0016
(login) and #0017 (recovery) build on.

- `internal/auth/webauthn.go` â€” `NewWebAuthn(cfg)` builds the relying party from
  `WEBAUTHN_RP_ID`/`WEBAUTHN_RP_ORIGIN`. `RegistrationUser` implements the
  `webauthn.User` interface: `WebAuthnID()` is a random 16-byte handle,
  name/displayName are the email, credentials empty, icon "". `registrationOptions()`
  centralizes the PRD passkey policy (residentKey required, userVerification
  required, attachment omitted, ES256 then RS256).
- `internal/auth/store.go` â€” data layer over the pgx pool. Reads
  `registrations_enabled` fresh on every call (never cached); manages
  pending_registrations, webauthn_challenges (challenge consumed via
  `DELETE ... RETURNING` for single-use), users (with admin promotion),
  passkey_credentials, and sessions. Accepts a `querier` so methods run on the
  pool or inside a tx.
- `internal/auth/registration.go` â€” `RegistrationService` with `StartRegistration`,
  `VerifyRegistration`, `FinishRegistration`. Finish runs in a single
  transaction (consume challenge â†’ verify attestation â†’ create user â†’ store
  credential â†’ delete pending row â†’ create session).
- `internal/auth/session.go` â€” `NewSessionToken()` (32 random bytes, URL-safe)
  and `SetSessionCookie()` (HttpOnly; Secure; SameSite=Strict; 30-day expiry).
  Shared by all auth flows.
- `internal/auth/tokens.go` â€” crypto-random byte/URL-token helpers.
- `internal/handlers/auth.go` â€” `AuthHandler` with RegisterStart/Verify/Finish,
  plus reusable `writeJSON`/`writeError`/`decodeJSON` helpers. Routes wired in
  `cmd/shortlinks/main.go`.

## Gotchas

Decisions the auth foundation rests on; #0016/#0017 should reuse them:

- **Session helper.** `auth.NewSessionToken()` + `auth.SetSessionCookie(w, token,
  expiresAt)` and `Store.CreateSession(ctx, q, userID, token, now)` are the
  single source of truth for session creation. Cookie name is
  `auth.SessionCookieName` (`shortlinks_session`). Login/recovery must mint
  sessions the same way so middleware (#0016) sees a consistent cookie.
- **WebAuthn user type.** `RegistrationUser` is for the pre-account registration
  ceremony (empty credential list, fresh random handle). #0016 needs a separate
  `webauthn.User` backed by the stored `passkey_credentials` (with
  `WebAuthnCredentials()` populated) for assertions; both satisfy the same
  interface and use the same `*webauthn.WebAuthn` from `auth.NewWebAuthn`.
- **Challenge handle is not persisted.** `webauthn_challenges` stores only the
  raw challenge bytes (the schema has no handle column).
  `FinishRegistration` only checks that `session.UserID == user.WebAuthnID()`,
  and both come from one `RegistrationUser`, so a fresh handle is regenerated on
  finish â€” the cryptographic binding is the challenge, not the handle. The
  challenge column is BYTEA; `session.Challenge` is base64url, so we
  decode-on-save and encode-on-load.
- **Challenge is single-use.** Consumed via `DELETE ... RETURNING` inside the
  finish transaction, preventing replay. Login challenges (#0016) should do the
  same.
- **`registrations_enabled` is read fresh every attempt** (never cached) per the
  PRD; do not add a cache.
- **No account-existence leak.** `StartRegistration` returns `ErrEmailRegistered`,
  but the handler maps it to the same 200 "Check your email" response as success.
- **AAGUID storage.** Stored as a UUID; an absent or all-zero AAGUID (common for
  synced platform passkeys) is written as SQL NULL by `aaguidArg`.
- **Audit writes deferred.** `account.registration_started`, `account.registered`,
  and `credential.added` are TODO(#0025) at the call sites; device_name and
  aaguid are already on the credential row so #0025 only needs to add the
  log writes.
