## Description

Implement the authenticated routes that let a user view, rename, and revoke their registered passkeys. Prevents the user from revoking their last credential.

## Acceptance Criteria

- [ ] `GET /account/credentials`: returns array of the authenticated user's passkeys with fields `{id, device_name, aaguid, created_at, last_used_at}`
- [ ] `PATCH /account/credentials/{id}`: updates `device_name`; returns `200` with updated credential; returns `404` if credential does not belong to authenticated user
- [ ] `DELETE /account/credentials/{id}`: revokes credential; returns `409 Conflict` with `{"error":"cannot_revoke_last_credential"}` if it is the user's only credential; writes `credential.revoked` audit entry; returns `200` on success
- [ ] All three routes require auth (use `RequireAuth` middleware from #0017)
- [ ] `go test ./internal/handlers/` covers: list credentials, rename, revoke non-last, revoke-last blocked

## Implementation

Added the data-layer queries to `internal/auth.Store` and a new
`internal/handlers.CredentialsHandler`, wired behind `middleware.RequireSession`
in `cmd/shortlinks/main.go`. Every store method takes the authenticated user id
and scopes its query to that user, so a request can only ever see or mutate its
own credentials.

- `GET /account/credentials` returns a JSON array of
  `{id, device_name, aaguid, device_hint, sign_count, created_at, last_used_at}`.
  The `public_key` and raw `credential_id` bytes are never selected by
  `ListCredentialsForUser`, so they cannot leak through this API. `device_hint`
  is derived from the AAGUID via a small known-authenticator map (e.g. the Apple
  AAGUID → "iCloud Keychain"); an absent/NULL AAGUID → "Unknown", an unrecognized
  one → the raw UUID string. An empty list is emitted as `[]`, never `null`.
- `PATCH /account/credentials/{id}` accepts `{device_name}` (or `{name}` as an
  alias), trims and rejects an empty name with `400`, and renames via a
  user-scoped `UPDATE`. A credential owned by another user (or absent) matches
  zero rows → `ErrCredentialNotFound` → `404` (no existence leak). Returns the
  updated record on success. Uses Go 1.22 `r.PathValue("id")`.
- `DELETE /account/credentials/{id}` revokes via `RevokeCredential`, which runs
  inside a transaction: it locks the user's rows with `FOR UPDATE`, tallies them
  and checks ownership in Go (an aggregate can't be combined with `FOR UPDATE`),
  returns `ErrCredentialNotFound` (→ `404`) for a foreign/absent id, and
  `ErrLastCredential` (→ `409 {"error":"cannot_revoke_last_credential"}`) when it
  is the account's only passkey. The transaction makes the count-then-delete
  atomic so two concurrent revokes can't both leave the account at zero.

The `credential.revoked` / `credential.added` audit-log writes are #0025 and are
intentionally left as a TODO/no-op (a comment marks the spot in `Revoke`).

## Gotchas

- PostgreSQL rejects `SELECT COUNT(*) … FOR UPDATE` (an aggregate can't carry a
  row lock). The first revoke implementation hit this and surfaced as a `500` in
  the tests; the fix selects the locked `id`s and tallies/checks ownership in Go.
- The repo's integration packages all share one test DB and clean up with
  `TRUNCATE … CASCADE`. Running `go test ./...` lets multiple packages hit the DB
  concurrently, which deadlocks and clobbers rows — this is pre-existing (the
  `auth` + `middleware` packages exhibit it without any #0019 code). Run the
  integration suite with `go test -p 1 ./...`, which passes cleanly.
