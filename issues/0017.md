## Description

Implement the session authentication middleware that validates the session cookie on every protected request, loads the user from the sessions table, and attaches the user to the request context. Unauthenticated requests receive 401.

## Acceptance Criteria

- [ ] `internal/middleware/auth.go` implements a `RequireAuth` middleware function
- [ ] Reads session token from the `HttpOnly` cookie
- [ ] Looks up session in `sessions` table; verifies `expires_at` is in the future
- [ ] Updates `sessions.last_seen_at` on every authenticated request (sliding window)
- [ ] Attaches `*User` to the request context via a typed context key
- [ ] Returns `401 Unauthorized` (JSON `{"error":"unauthenticated"}`) if cookie is absent, token not found, or session expired
- [ ] `RequireAdmin` middleware wraps `RequireAuth` and additionally checks `users.is_admin = true`; returns `403 Forbidden` if not admin
- [ ] Helper `UserFromContext(ctx) (*User, bool)` extracts the user from context
- [ ] Routes requiring auth: all `/api/*` except `/api/events` auth handled inline; all `/account/*`; all `/admin/*`
- [ ] `go test ./internal/middleware/` covers: valid session, expired session, missing cookie, non-admin accessing admin route

## Notes

Dependencies: #0013 (sessions table), #0016 (login creates sessions).

## Implementation

The session guard lives in `internal/middleware/auth.go` (package `middleware`).
The original acceptance criteria named the function `RequireAuth`; it is
implemented as `RequireSession` to match the naming the dependent issues
(#0019/#0022/#0027/#0028) and the PRD ("session middleware") expect. Same
behaviour, clearer name.

Exported context API other handlers will use:

- `type AuthUser struct { ID int64; Email string; IsAdmin bool }` — the
  authenticated principal, defined in the `middleware` package so any handler
  package can read it without an import cycle with `auth`.
- `func UserFromContext(ctx context.Context) (*AuthUser, bool)` — retrieves the
  user attached by `RequireSession`.
- `func RequireSession(resolver SessionResolver) func(http.Handler) http.Handler`
  — the guard. Reads the `shortlinks_session` cookie; **401** (JSON
  `{"error":"unauthenticated"}`) when absent, when the token is unknown/expired,
  or when the owning account is deactivated (`active = false`). On success it
  attaches `*AuthUser` to the context and calls `next`.
- `func RequireAdmin(next http.Handler) http.Handler` — runs after
  `RequireSession`; **403** (`{"error":"forbidden"}`) for non-admins, **401** if
  no session is on the context. Admin routes wrap as
  `RequireSession(store)(RequireAdmin(handler))`.
- `type SessionResolver interface { ResolveSession(ctx, token, now) (auth.SessionUser, error) }`
  — the single data-layer dependency, satisfied by `*auth.Store`. Taking an
  interface keeps the guard fakeable and decoupled.

Data access is centralised in `auth.Store.ResolveSession` (new) rather than
duplicated in the middleware. In one SQL round-trip it joins `sessions` → `users`
by the raw cookie value, rejects an expired session (`expires_at <= now`) or a
deactivated owner, and — on success — bumps `last_seen_at = now` and extends
`expires_at = now + 30d` (the sliding window). The `UPDATE ... RETURNING` re-checks
`expires_at` in its `WHERE` so a row expiring between read and write is not
revived. On no match it does a follow-up read to distinguish expired
(`ErrSessionInvalid`, and best-effort deletes the dead row) from inactive
(`ErrAccountInactive`); both map to 401 in the middleware so no information leaks.

## Gotchas

- **HMAC discrepancy (follow-up needed).** PRD line 737 (env vars) says
  `SESSION_SECRET` should HMAC-sign the session token in the cookie. The existing
  scheme from #0015/#0016 (`NewSessionToken` / `SetSessionCookie`) stores a raw
  random 32-byte token directly as the cookie value, and PRD line 542 (Session
  Security) describes exactly that raw scheme. To avoid half-implementing a
  parallel scheme and breaking #0015/#0016, the guard validates against the
  existing raw-token scheme (direct DB lookup of the cookie value). If
  HMAC-signed cookies are required, it should be a dedicated follow-up that
  updates the mint path (#0016) and this lookup together, and reconciles the two
  conflicting PRD lines.
- **Acceptance criterion rename.** `RequireAuth` in the criteria → `RequireSession`
  in code (noted above). Wiring the actual route table (`/api/*` except
  `/api/events`, `/account/*`, `/admin/*`) is the router's job and belongs to the
  handler/routing issues that consume this guard, not to the middleware package.
- **Why the two-step resolve.** The `UPDATE ... RETURNING` cannot tell "no such
  token" from "expired" from "inactive" — they all produce zero rows. A single
  follow-up `SELECT` classifies the failure so 401 vs (internal) inactive is
  precise and the expired row can be deleted; live traffic on valid sessions
  still takes just the one `UPDATE`.
