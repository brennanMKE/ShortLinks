## Description

The recovery backend (`/auth/recover`, `/auth/recover/verify`, `/auth/recover/finish`) was built in #0018, but no frontend was ever scoped for it. The Login view has no "lost passkey / recover account" affordance, there is no recover sub-form, and there is no recovery landing view to run the credential-creation ceremony. The only mention of recovery in `web/src` is a comment in `Account.svelte`. The PRD's Account Recovery flow (lines 295–309) is unreachable from the UI.

## Long Description

Recovery is the intended path for "account exists but all passkeys are lost" — including the case of an account that has never had a passkey (see #0044). `StartRecovery` adds a credential to the existing account without creating a new user (`recovery.go:64-106`), so it is the correct mechanism for enrolling a passkey onto an already-existing account. But with no UI, it cannot be triggered or completed in a browser today.

This shares the magic-link landing mechanism (#0041) and the credential-creation helpers (#0042); build on both rather than duplicating.

## Expected behavior

A "Recover account" link on the login page opens an email form that posts to `/auth/recover`; the recovery email's link lands on a page that runs the passkey-creation ceremony and attaches the new credential to the existing account.

## Actual behavior

No recovery entry point exists in the SPA; the backend routes are unreachable from the UI.

## Root cause

The recovery backend (#0018) shipped with no frontend: no entry point on Login, no landing view ceremony. The PRD's recovery flow was unreachable from the browser.

## Fix

Added `recoverStart` / `recoverFinish` API wrappers (finish: token + `device_name` as query params, attestation as JSON body, returns `{user_id}`), completed `RecoverVerify.svelte` to drive verify → `create()` → finish → `getMe()` → navigate (reusing `toPublicKeyCreationOptions`/`serializeAttestation` from #0042 verbatim), and added a "Recover account / lost passkey" sub-form to `Login.svelte` mirroring the existing Register sub-form (generic, no-enumeration confirmation copy).

## Review

## Gotchas

Recover finish returns `{user_id}` (not the user object register returns), so the view MUST call `/api/me` to populate `currentUser`. Otherwise the contract is identical to register finish.
