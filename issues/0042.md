## Description

The Svelte SPA can *start* registration (the Login view's Register sub-form posts the email to `/auth/register/start`) but cannot *finish* it. There is no magic-link landing view, no `navigator.credentials.create()` call, no wiring to `GET /auth/register/verify` or `POST /auth/register/finish`, and `web/src/lib/webauthn.ts` only implements the login assertion (`.get()`) path — there are no credential-*creation* helpers. The PRD's Account Creation Flow (lines 214–245) is therefore only half-built: backend done (#0015), frontend missing.

## Long Description

Issue #0032 (Login view) scoped the register affordance only as far as *"→ displays 'Check your email' message"* — the magic-link landing, the attestation ceremony, and the `verify`/`finish` calls were never in any frontend ticket, so they were never built. Consequences confirmed in code:

- `web/src/lib/webauthn.ts` exports only `toPublicKeyRequestOptions` / `serializeAssertion` (login). No `toPublicKeyCreationOptions` / `serializeAttestation`.
- No `navigator.credentials.create()` anywhere in `web/src`.
- No API wrappers for `register/verify` or `register/finish` in `web/src/lib/api.ts` (only `registerStart`).
- `App.svelte` has no view or route for a verification landing.

Depends on #0041 for how the token-bearing URL reaches the SPA.

## Expected behavior

After clicking the magic link, the user sees a passkey prompt, completes it, and lands authenticated on the dashboard — including correct admin promotion for the `ADMIN_EMAIL` account (`FinishRegistration` already handles this at `registration.go:215`).

## Actual behavior

The magic link goes nowhere usable (#0041); even if it did, the SPA has no code to fetch creation options, call `create()`, or post the attestation.

## Resolution notes

> 🟢 Resolved 2026-06-17 — The SPA now completes registration: the landing view fetches creation options, runs `navigator.credentials.create()`, serializes the attestation, posts to `/auth/register/finish`, and lands the (possibly admin) user on the dashboard. ⚠️ Verified by build + 26 unit tests + Opus review of the wire format; a live on-device passkey ceremony has NOT been exercised and remains the confirmation step before closing.

## Root cause

Issue #0032 scoped the Login view's Register button only as far as "Check your email." The magic-link landing, the `navigator.credentials.create()` attestation ceremony, and the `verify`/`finish` calls were never built; `webauthn.ts` had only the login (assertion/`.get()`) helpers.

## Fix

Added the attestation parallel to the existing assertion helpers in `webauthn.ts` (`ServerCredentialCreation` type, `toPublicKeyCreationOptions`, `serializeAttestation`), an `api.ts` `registerFinish` wrapper (token + `device_name` as query params, attestation as JSON body, matching the backend), and completed `RegisterVerify.svelte` to drive verify → `create()` → finish → `getMe()` → set `currentUser` → navigate to dashboard, with inline error states. Admin promotion is server-side in `FinishRegistration`; the Admin tab is gated on `currentUser.is_admin`, identical to the login path.

## Gotchas

`device_name` is a QUERY param on `/auth/register/finish`, not a body field (the body is the bare attestation JSON) — same for recovery (#0043). go-webauthn serializes `user.id` as base64url even though its Go type is `any`; the frontend must base64url-decode it.
