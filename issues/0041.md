# 0041 — Magic-link verify URLs return raw JSON instead of an enrollment page

| | |
|---|---|
| **Status** | open |
| **Module** | handlers / web / auth |
| **Platform** | All |
| **First seen** | 2026-06-17 |

## Description

The registration and recovery emails link to `{BASE_URL}/auth/register/verify?token=…` and `{BASE_URL}/auth/recover/verify?token=…` (`internal/auth/mailer.go:46,51`), but those paths are registered as JSON API handlers (`main.go:178,186`) that return WebAuthn `PublicKeyCredentialCreationOptions`. A browser following the link from an email therefore renders a raw JSON blob — there is no HTML/JS to read the token and run the passkey ceremony. This blocks every email-driven enrollment flow.

## Long Description

The SPA is store-driven with no client-side router and no code that reads a `?token=` query param (confirmed: nothing in `web/src` references `location.search`, `searchParams`, `register/verify`, or `recover/verify`). The catch-all SPA handler is served last (`main.go:244`), but the explicit `GET /auth/*/verify` routes shadow it, so even a deep-link fallback never reaches `index.html`.

A coherent fix needs a decision on the URL scheme. Two viable approaches:
1. Point the email links at an SPA-served path (e.g. `/register?token=…`, `/recover?token=…`); the SPA reads the token on load and calls a JSON endpoint to fetch the creation options.
2. Keep the email link on `/auth/*/verify` but have that GET serve the SPA shell (HTML), and move the JSON options to a separate path the SPA fetches.

This is the shared foundation for #0042 (registration completion UI) and #0043 (recovery UI).

## Expected behavior

Clicking a magic link in an email lands on a page that loads the SPA, reads the token, and begins the passkey ceremony.

## Actual behavior

The browser displays raw JSON WebAuthn options; nothing runs.

## Acceptance Criteria

- [ ] Decide and document the magic-link URL scheme (SPA route vs. HTML-serving verify endpoint)
- [ ] Email links produced by `verificationURL`/`recoveryURL` resolve to a browser page that loads the SPA, not a JSON response
- [ ] The SPA reads the `token` from the landing URL and makes it available to the registration/recovery ceremony
- [ ] The JSON creation-options payload is reachable by the SPA via a documented endpoint (renamed/relocated if needed) without colliding with the browser-facing link
- [ ] A hard refresh on the landing URL still works (SPA fallback to `index.html`)
- [ ] `go build ./...` and `cd web && npm run build` pass
