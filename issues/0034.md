## Description

Add real-time link list updates to the Dashboard view using Server-Sent Events. When a new link is created (or an inactive link is reactivated), it appears at the top of the list instantly without polling.

## Acceptance Criteria

- [ ] `web/src/lib/events.ts` opens `new EventSource('/api/events')` and listens for `link.created` messages
- [ ] On `link.created` event, the new link object is prepended to the `links` Svelte store (list updates immediately)
- [ ] `EventSource` connection is opened when the Dashboard view mounts and closed (`.close()`) when it unmounts
- [ ] If the SSE connection drops, the browser's built-in `EventSource` reconnection handles recovery (no custom retry logic needed)
- [ ] Deduplication: if the `link.created` event carries a key that is already in the `links` store (e.g., from the creation form response), it replaces rather than duplicates the entry
- [ ] End-to-end test: create a link via the form in one browser tab, confirm it appears in the list in a second tab without a page refresh

## Implementation

Added `web/src/lib/events.ts`, an SSE client:

- `subscribeLinks(onCreated, factory?)` opens `new EventSource('/api/events')`,
  attaches a `link.created` listener that `JSON.parse`s each frame's `data`
  into a `Link` and invokes `onCreated`, and returns a cleanup function that
  calls `.close()`. The EventSource is injectable (default = the real global)
  purely so tests can pass a fake.
- A malformed frame is swallowed (parse error → `return`), so one bad event
  cannot crash the dashboard or end the stream.
- `prependUniqueByKey(list, link)` is the pure, immutable store policy: it
  prepends the incoming link and drops any existing entry with the same key, so
  a duplicate key REPLACES its row at the front rather than double-inserting.
- No custom reconnect/backoff — the browser's built-in EventSource reconnection
  handles drops, as the AC specifies.

Wired into `web/src/views/Dashboard.svelte` at the `#0034` seam: `onMount` now
calls `subscribeLinks`, updating the `links` store via `prependUniqueByKey`, and
returns the cleanup function so the stream closes on unmount. The seam comment
and the stale "SSE is NOT implemented here" header note were removed/updated.

The server side was confirmed, not changed: `internal/handlers/events.go`
writes `event: link.created\ndata: <JSON>\n\n` frames, and
`internal/handlers/links.go` publishes `events.Event{Name:"link.created",
Payload: json.Marshal(linkView)}` — the same `linkView` the create response
returns — so `JSON.parse(e.data)` yields a `Link`.

## Gotchas

The user's own creates are prepended optimistically in `handleCreate` (#0033)
AND also arrive back over SSE (the server publishes `link.created` for the
creating user too). Without dedupe the link would appear twice. The fix is the
key-based dedupe in `prependUniqueByKey`: the SSE arrival replaces the existing
row (moving it to the front with the server's authoritative data) instead of
inserting a second copy. The same helper is used for foreign creates (new key →
simple prepend).
