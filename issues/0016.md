## Description

Implement the WebAuthn login flow with conditional UI (passkey autofill) support, sign_count handling for iCloud Keychain synced passkeys, and session issuance.

## Acceptance Criteria

- [ ] `GET /auth/login/start`: issues `PublicKeyCredentialRequestOptions`; if `?email=` query param is provided, populates `allowCredentials` filtered to that user's credentials; stores challenge in `webauthn_challenges`; returns options JSON
- [ ] `POST /auth/login/finish`: finds credential by `credential_id`; verifies signature against stored public key; checks `users.active = true` (returns `403 Account deactivated` if not); handles sign_count per PRD spec:
  - Stored `sign_count = 0` and assertion returns `0`: accept (iCloud Keychain behavior)
  - Stored `sign_count > 0` and assertion ≤ stored: log warning (possible clone)
  - Otherwise: update stored `sign_count`
- [ ] On success: updates `users.last_login_at`; creates `sessions` row (30-day expiry); sets `HttpOnly; Secure; SameSite=Strict` cookie
- [ ] `POST /auth/logout`: deletes session row; clears cookie; returns `200`
- [ ] Writes `account.login` audit entry on successful login, `account.logout` on logout
- [ ] `go test ./internal/handlers/` covers: successful login, deactivated user, invalid challenge, sign_count warning case

## Implementation

Built on the #0015 foundation — reused the shared `auth.Store` (extended, not
forked), `NewWebAuthn`, `NewSessionToken`/`SetSessionCookie`, and the
`handlers.AuthHandler` JSON helpers.

- **`LoginUser`** (`internal/auth/webauthn.go`): a second `webauthn.User`
  implementation backed by stored `passkey_credentials`. `WebAuthnCredentials()`
  returns the account's real passkeys so go-webauthn matches the incoming
  assertion against them.
- **`LoginService`** (`internal/auth/login.go`):
  - `StartLogin(email)` → `BeginLogin` with `allowCredentials` when the email
    maps to an existing account with credentials, else `BeginDiscoverableLogin`
    (conditional UI). The challenge is persisted to `webauthn_challenges`
    (purpose `authentication`, 5-min TTL). The response is identical whether or
    not the email exists, so account existence never leaks (lookup errors are
    swallowed to "no user").
  - `FinishLogin(r)` → parses the assertion, resolves the credential **and**
    account by the assertion's credential id (covers both discoverable and
    non-discoverable login since rawID is always present), enforces
    `users.active` returning `ErrAccountDeactivated` **before** any session is
    issued, consumes the single-use challenge inside a transaction
    (`DELETE … RETURNING`), verifies via `ValidateLogin`, applies the sign_count
    rules, sets `last_login_at`/`last_used_at`, and creates the session.
  - `Logout(token)` → idempotent `DELETE FROM sessions`.
- **sign_count** (PRD): go-webauthn's `UpdateCounter` raises `CloneWarning` on a
  regression. Mapped onto storage: clone case → warn-log + leave stored counter
  unchanged; stored 0 / assertion 0 → accept silently, leave at 0; otherwise
  store the advanced counter.
- **Handlers** (`internal/handlers/auth.go`): `LoginStart` (200 + options),
  `LoginFinish` (200 + cookie / 403 `Account deactivated` / generic 401),
  `Logout` (200, expires the cookie). Routes wired in `cmd/shortlinks/main.go`.
- `account.login` / `account.logout` audit writes are left as `TODO(#0025)`.

## Gotchas

- **No user-handle column.** The schema (v7) persists no WebAuthn user handle
  (not on `users`, not on `passkey_credentials`). So login resolves the account
  by **credential id** (the assertion's rawID), and `LoginUser`'s handle is
  mirrored from the assertion's own userHandle. The real proof of possession is
  the signature verified against the stored public key — the handle echo just
  keeps go-webauthn's `userHandle == WebAuthnID` check consistent. If a future
  issue wants stricter discoverable-login handle binding, it must add a handle
  column + migration.
- **Per-session challenge isn't stored**, so `session.Challenge` is set from the
  assertion's own client-data challenge; `ValidateLogin`'s challenge comparison
  is therefore trivially satisfied. The binding to a server-issued challenge is
  enforced by the DB `ConsumeAuthenticationChallenge` step (single-use, TTL) —
  an assertion whose challenge was never issued (or already consumed/expired)
  fails there.
- **Backup flags.** The schema doesn't persist BE/BS flags; reconstructed
  credentials use the zero `CredentialFlags`. This matches both the
  `virtualwebauthn` default and the credentials this RP itself registered, so
  go-webauthn's BE-consistency check passes.
