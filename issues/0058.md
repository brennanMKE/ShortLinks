# 0058 — Dev-only auth bypass with a mock admin user (skip the passkey ceremony on localhost)

| | |
|---|---|
| **Status** | open |
| **Module** | auth |
| **Platform** | macOS |
| **First seen** | 2026-06-22 |

## Description

Add a strictly dev-only mode that establishes an authenticated session for a mock admin user **without** the WebAuthn passkey ceremony, so opening `localhost` lands directly on the dashboard. WebAuthn requires HTTPS and a real authenticator, which is impractical for quick local checks.

## Motivation

Even with a local server, the SPA's first reachable view is login, and the passkey ceremony can't be completed casually on localhost. A dev auth bypass makes the authenticated UI (dashboard / account / admin) reachable for inspection. (Part of [#0056](0056.md).)

## Proposed approach

- Behind an explicit dev flag (e.g. `DEV_AUTH=true`), seed/load a mock admin user in the active store (#0057) and issue a valid session cookie automatically — e.g. an auto-login endpoint, or middleware that establishes the session on first request.
- **Hard guardrail:** refuse to enable unless clearly non-production — require `DEV_AUTH=true` AND a localhost `BASE_URL` / `WEBAUTHN_RP_ID`, and/or a `dev` build tag. Fail closed; never bypass auth in production. Log loudly when dev auth is active.
- Keep the real WebAuthn login/registration/recovery flows intact and the default; this is additive and off by default.

## Acceptance criteria

- [ ] With the dev flag on (localhost only), loading the app lands on the dashboard as a mock admin with a valid session — no passkey
- [ ] The bypass is impossible to enable in production (explicit flag + non-prod/localhost check and/or build tag); covered by a test
- [ ] Real passkey login/registration/recovery flows are unchanged when the flag is off
- [ ] `go build ./...` and `go test ./...` pass
