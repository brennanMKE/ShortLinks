# 0052 — Elevate the SPA's visual and UX design beyond the minimal baseline

| | |
|---|---|
| **Status** | open |
| **Module** | web |
| **Platform** | All |
| **First seen** | 2026-06-18 |

## Description

The management SPA is functional but visually minimal — plain cards, default-ish typography, ad-hoc spacing, and little visual hierarchy or polish. Building on the responsive layout work (#0050) and the light/dark theming work (#0051), give the SPA a deliberate, cohesive design: a defined visual language (type scale, spacing rhythm, color, component styling, interactive states) that makes the app feel considered and on-brand rather than a bare scaffold.

## Motivation

This is the public-facing management UI at `go.sstools.co`, reached from any device. A more refined design improves usability, trust, and perceived product quality. #0050 establishes the structural (responsive) groundwork and #0051 establishes the theming token groundwork; this issue is the layer on top — hierarchy, rhythm, component consistency, states, and overall polish — so the three together turn a minimal scaffold into a finished-feeling app.

## Proposed approach

> Initial frame only — the user will provide specific design direction, so scope and specifics below are a starting point to be refined.

- Establish a small design language: type scale, spacing scale, corner radii, elevation/shadows — sourcing color from the #0051 theme tokens so light/dark stay correct.
- Componentize the repeating UI (buttons, inputs, cards, tables, badges, empty / loading / error states) for visual consistency across views.
- Improve visual hierarchy per view (`Login`, `Dashboard`, `LinkDetail`, `Account`, `Admin`, `RegisterVerify`, `RecoverVerify`) — clear headings, grouping, and whitespace.
- Refine interactive states (hover / focus / active / disabled), transitions, and user feedback (notices/toasts).
- Keep it lightweight — stay in Svelte 5 with no heavy UI framework unless the user wants one; honor the embed-in-Go-binary constraint and the existing Vite build.

## Acceptance criteria

- [ ] A documented set of design primitives (type, spacing, radii, shadows; color via #0051 tokens)
- [ ] Consistent, reusable component styling across all views
- [ ] Clear visual hierarchy and spacing rhythm; the UI feels cohesive rather than ad hoc
- [ ] Interactive states and feedback are consistent and polished in both light and dark mode
- [ ] No regression to responsiveness (#0050) or theming (#0051); passkey ceremonies still work
- [ ] `cd web && npm run build` and `npm run check` clean
