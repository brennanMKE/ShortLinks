# 0076 — Expires date field can no longer be cleared back to empty (regression from #0053)

| | |
|---|---|
| **Status** | open |
| **Module** | web |
| **First seen** | 2026-06-24 |

## Description

The **Expires (optional)** field on the create-link form can no longer be reset to empty once a date is set. Expiry is optional, so the user must be able to clear it — but there is now no way to do so.

## Steps to reproduce

1. On the Dashboard "Create a short link" form, set an Expires date/time.
2. Try to clear it back to empty (no expiry).
3. There is no clear / "none" affordance — the value can't be removed.

## Expected behavior

An optional date field can be cleared back to empty, so the link is created with no expiry.

## Actual behavior

Once set, Expires can't be cleared.

## Notes

**Regression introduced by #0053.** To fix the field's width on iOS Safari, #0053 added `-webkit-appearance: none` to date/time inputs in `web/src/app.css`, which strips the native `datetime-local` control's chrome — including its built-in clear button. On iOS the picker also has no "none" option, so the value becomes unclearable.

The create logic already treats an empty `expiresAt` as "no expiry" (`web/src/views/Dashboard.svelte:132-136` — empty string ⇒ no `expires_at` sent), and `expiresAt` resets to `''` after a successful create. So the data side is fine; only the **UI affordance to clear** is missing.

**Fix approach (do NOT regress width/height):** add an explicit **Clear** control next to the Expires input that sets `expiresAt = ''` (shown only when a value is present). Do **not** revert the #0053 width fixes (`min-width: 0`, `-webkit-appearance: none`) or the height fixes (`min-height`, `line-height`) in `app.css` — those stay. The Expires field is at `web/src/views/Dashboard.svelte:322`.

## Root cause

#0053 added `-webkit-appearance: none` to the `datetime-local` input (to fix its width on iOS Safari), which strips the native control's built-in clear button. The iOS picker has no "none" option, so once a date was set the optional Expires field could not be returned to empty.

## Fix

Added a small "Clear expiry date" text link **below** the Expires input (`web/src/views/Dashboard.svelte`), rendered only when `expiresAt !== ''`, that sets `expiresAt = ''`. It is `type="button"` (never submits) and clears **only** `expiresAt` — not the rest of the form. Crucially, the `<input>` element itself is left **byte-identical to `main`** (it was deliberately NOT wrapped or resized), and `app.css` is **not touched at all**, so the #0053 width (`min-width:0`, `-webkit-appearance:none`) and height (`min-height`, `line-height`) fixes are fully intact — no width/height change. Empty `expiresAt` already means no `expires_at` is sent (`buildInput:132-136`), so a cleared field creates a link with no expiry.

(An initial attempt put the × button inline beside the input via a flex row, which slightly narrowed the input; that was discarded in favor of the below-input link so the input's geometry stays provably unchanged.)

## Review notes (2026-06-24)

Brutally honest re-review because the user reports the Expires clear is not working / not what they wanted.

**What IS working (deployment/serving is fine — "not deployed" is NOT the cause):**
- The live JS bundle contains the string `Clear expiry date` (`curl -s https://go.sstools.co/assets/index-BGpuarXL.js | grep "Clear expiry date"` → hit). The code is deployed.
- The markup is logically sound: `web/src/views/Dashboard.svelte:330-337` renders a `type="button"` (never submits) with `onclick={() => { expiresAt = ''; }}`, gated on `{#if expiresAt !== ''}`. It clears only `expiresAt`, and `buildInput` (lines 132-136) omits `expires_at` when empty, so a cleared field genuinely creates a no-expiry link. The data path is correct.

**What was NOT done properly:**
1. **The clear was never verified on a device/browser — and there is real reason to doubt it on iOS Safari.** The state `expiresAt = ''` is the Svelte source of truth and `bind:value` is two-way, so in principle setting it to `''` should empty the control. BUT `datetime-local` with `-webkit-appearance: none` (kept from #0053) is exactly the configuration that renders unreliably on iOS Safari, and the *visible* rendered value of a stripped-chrome `datetime-local` does not always track a programmatic empty-string write the way a normal text input does. This was **explicitly not tested on a device** — the Verification section says the behavior is "verified by logic" and the "visual look/placement ... on a real iPhone is the user's sign-off." So the one environment where #0053 broke things (iOS Safari) is the one environment never re-checked. If the user is on iPhone and the field doesn't visibly empty, that is a plausible real bug, not just caching.
2. **The fix does not meet the user's actual expectation.** The user said "before there was a reset button for that field" — they expect an **in-field reset affordance** (the native clear button that #0053 stripped). What shipped is a small text link *below* the input, only visible after a value is set, styled as faint accent-colored text with no icon. That is easy to miss: it is below the field (not where a clear control conventionally lives), low-contrast, and visually reads like a caption rather than a control. It technically restores the capability but does not restore the *affordance* the user remembers, which is why they may not even realize it is the fix.
3. **Marked `resolved`/`closed` with zero browser confirmation that the link even appears or clears.** Same gap as #0061: closed on `npm run check`/`build`/`test` (which exercise none of this UI behavior) plus logic reasoning, with the real check deferred to the user. The user is now the one finding it doesn't meet the need.

**Most likely reason the user "doesn't see it working":** two possibilities, both real — (a) they are looking for an in-field reset button and don't recognize the small below-field text link as the clear control (discoverability failure), and/or (b) on iOS Safari the programmatic `expiresAt = ''` does not visibly empty the `-webkit-appearance:none` `datetime-local`, so even when they tap the link the field still shows the old date. The string is deployed; the deficiency is UX + an unverified iOS rendering assumption.

**To make it genuinely done:**
1. Test on a real iPhone (Safari): set a date, tap "Clear expiry date", and confirm the field visibly empties *and* a created link has no expiry. If the field doesn't visually clear, force it (e.g. briefly toggle the input via a keyed re-render, or reset `input.value` directly) — don't rely solely on the bound empty string.
2. Reconsider the affordance to match the user's mental model: put a visible in-field/adjacent clear control (an "×" inside or immediately right of the input) instead of a faint link below it — while still not regressing #0053's width (the prior inline-× attempt narrowed the input; solve that with absolute positioning over the input's right edge rather than a flex row).
3. Increase discoverability: clearer label/icon and contrast so it doesn't read as a caption.
4. Only mark closed after on-device confirmation, not after logic-only verification.
