# 0076 — Expires date field can no longer be cleared back to empty (regression from #0053)

| | |
|---|---|
| **Status** | open |
| **Module** | web |
| **First seen** | 2026-06-24 |

## Description

The **Expires (optional)** field on the create-link form can no longer be reset to empty once a date is set. Expiry is optional, so the user must be able to clear it — but there is now no way to do so.

## Steps to reproduce

1. On the Dashboard "Create a short link" form, set an Expires date/time.
2. Try to clear it back to empty (no expiry).
3. There is no clear / "none" affordance — the value can't be removed.

## Expected behavior

An optional date field can be cleared back to empty, so the link is created with no expiry.

## Actual behavior

Once set, Expires can't be cleared.

## Notes

**Regression introduced by #0053.** To fix the field's width on iOS Safari, #0053 added `-webkit-appearance: none` to date/time inputs in `web/src/app.css`, which strips the native `datetime-local` control's chrome — including its built-in clear button. On iOS the picker also has no "none" option, so the value becomes unclearable.

The create logic already treats an empty `expiresAt` as "no expiry" (`web/src/views/Dashboard.svelte:132-136` — empty string ⇒ no `expires_at` sent), and `expiresAt` resets to `''` after a successful create. So the data side is fine; only the **UI affordance to clear** is missing.

**Fix approach (do NOT regress width/height):** add an explicit **Clear** control next to the Expires input that sets `expiresAt = ''` (shown only when a value is present). Do **not** revert the #0053 width fixes (`min-width: 0`, `-webkit-appearance: none`) or the height fixes (`min-height`, `line-height`) in `app.css` — those stay. The Expires field is at `web/src/views/Dashboard.svelte:322`.
