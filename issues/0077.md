# 0077 — Support iPhone-app authentication via Authorization: Bearer (reuse passkey session token)

| | |
|---|---|
| **Status** | open |
| **Module** | auth / handlers |
| **Platform** | iOS · All |
| **First seen** | 2026-06-24 |

## Description

Enable a native iPhone app to authenticate against the existing REST API
(`/api/*`) without inventing a new auth system. Today every `/api/*` route is
guarded by `RequireSession`, which reads the session **cookie**
(`shortlinks_session`, `HttpOnly; Secure; SameSite=Strict`) and looks the opaque
token up in the DB (`internal/middleware/auth.go`, `internal/auth/session.go`).
That model is built for the browser SPA; there is no header-based credential for
a programmatic/native client.

**Option A (chosen):** keep the WebAuthn/passkey login ceremony — iOS supports
passkeys natively via `AuthenticationServices` — and let the same session token
be presented in an `Authorization: Bearer <token>` header **in addition to** the
cookie. The app runs `GET /auth/login/start` → native passkey assertion →
`POST /auth/login/finish`, receives the session token, and sends it as a bearer
header on subsequent `/api/*` calls. No passwords, no second token system, and
the validation path (DB lookup + 30-day sliding window) is unchanged.

This was chosen over Option B (separate personal-access-tokens / API keys) and
Option C (OAuth2/OIDC) as the smallest, most secure first step that reuses the
already-tested auth. Option B may follow later if headless/CLI access is wanted.

## Motivation

A first-party iPhone app needs to call the link CRUD API
(`POST/GET/PATCH/DELETE /api/links`, `GET /api/me`, `GET /api/events`). Relying
on the browser cookie from a native client is awkward (cookie-jar handling, and
`SameSite=Strict`/`HttpOnly` are browser-oriented). A bearer header is the
conventional, explicit way for an app to carry its credential, and it decouples
API clients from cookie semantics while keeping a single source of truth for the
token.

## Proposed approach

- **`RequireSession` accepts a bearer token as well as the cookie.** Read
  `Authorization: Bearer <token>`; if present, use that token, otherwise fall
  back to `r.Cookie(auth.SessionCookieName)`. Everything downstream
  (`ResolveSession`, the sliding-window bump, the deactivated-account check, the
  401 JSON response) stays identical. Header takes precedence over cookie when
  both are present.
- **No change to how tokens are minted.** `POST /auth/login/finish` (and
  registration/recovery) already return a session and `Set-Cookie`. Decide
  whether to *also* return the raw token in the JSON body so a native client can
  capture it without scraping `Set-Cookie` (it currently only lands in the
  cookie). Likely yes — add the token (and its `expires_at`) to the finish
  response body, guarded so the web SPA simply ignores it.
- **Passkey RP setup for iOS (app-side + ops, not server logic):** set
  `WEBAUTHN_RP_ID` to the production host (e.g. `go.sstools.co`, not
  `localhost`), add the app's **Associated Domains** entitlement, and serve an
  `apple-app-site-association` file so the app counts as the same Relying Party
  as the website and one passkey works in both.
- **Tests:** extend the `RequireSession` tests to cover (a) valid bearer token →
  authenticated, (b) bearer takes precedence over cookie, (c) malformed/empty
  `Authorization` header → falls back to cookie, (d) unknown bearer token → 401.

## Acceptance criteria

- [ ] `RequireSession` authenticates a request carrying a valid session token in
      `Authorization: Bearer <token>` with no cookie present.
- [ ] When both a bearer header and a cookie are sent, the bearer header wins.
- [ ] A malformed/empty `Authorization` header falls back to the cookie (no
      regression for the browser SPA).
- [ ] An unknown/expired bearer token returns the same 401 JSON as a bad cookie.
- [ ] The 30-day sliding-window bump and deactivated-account rejection behave
      identically regardless of which transport carried the token.
- [ ] `go build ./...` and `go test ./internal/middleware/... ./internal/handlers/...`
      pass, including new bearer-path tests.

## Out of scope

- Personal access tokens / API keys as a distinct credential type (Option B) —
  track separately if/when headless or CLI access is needed.
- OAuth2/OIDC for third-party apps (Option C).
- The iOS app itself (entitlements, AASA hosting, passkey UI) — this issue is
  the **server** enablement; app/ops work is noted for context only.

## Notes

- Security note: presenting the token in a header is no weaker than the cookie —
  it's the same opaque 32-byte DB-backed token with the same expiry. `HttpOnly`
  only constrains browser JS, not native HTTP clients, so a native app could
  technically reuse the cookie; the bearer path is about ergonomics and clarity,
  not a new trust boundary.
- Relevant code: `internal/middleware/auth.go` (`RequireSession`),
  `internal/auth/session.go` (`SessionCookieName`, token minting),
  `internal/handlers/auth.go` (login/finish response shape),
  `cmd/shortlinks/main.go:342-354` (the guarded `/api/*` routes).
- Related: builds on the existing WebAuthn work in `internal/auth/` (login,
  registration, recovery, webauthn).
