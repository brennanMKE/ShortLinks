## Description

Make the SPA adapt to light and dark mode. There is currently no theming infrastructure — no global stylesheet, no CSS custom properties, and no `prefers-color-scheme` handling; every component carries inline `<style>` with hardcoded hex colors (e.g. `#1f6feb`, `#c0362c`, `#666`). Introduce a small set of theme tokens and make the UI honor the OS color scheme.

## Motivation

A hardcoded light palette looks broken in dark-mode contexts (especially on phones at night) and is inconsistent across components. Centralizing colors also makes the responsive work and future visual changes far easier.

## Proposed approach

- Define a palette as CSS custom properties (`--color-bg`, `--color-surface`, `--color-text`, `--color-muted`, `--color-accent`, `--color-danger`, `--color-border`, …) on `:root`, with a dark override under `@media (prefers-color-scheme: dark)` (and/or a `[data-theme]` attribute if an explicit toggle is wanted).
- Introduce one global stylesheet (the SPA has none today) for the tokens and base element styling; migrate the hardcoded hex values in each component's inline `<style>` to the variables.
- Decide whether to ship an explicit light/dark/system toggle (persisted to `localStorage`) or follow the OS only — recommend OS-follow first, toggle as a possible follow-up.
- Set `<meta name="color-scheme">` / the CSS `color-scheme` property so form controls and scrollbars theme correctly.
- Verify contrast (WCAG AA) for text and interactive elements in both themes, including the error/notice colors.

## Acceptance criteria

- [x] Theme tokens defined as CSS variables with a `prefers-color-scheme: dark` override
- [x] All views legible and on-brand in both light and dark mode (no hardcoded hex left that breaks in dark)
- [x] Native form controls/scrollbars respect the active scheme
- [x] Text + interactive contrast meets WCAG AA in both themes
- [x] `cd web && npm run build` and `npm run check` clean

## Implementation

Added OS-driven light/dark mode by overriding token *values*, not components. A `@media (prefers-color-scheme: dark) { :root { … } }` block in `web/src/app.css` restates all 18 color tokens with a GitHub-Primer-style dark palette (near-black `#0d1117` page / `#161b22` panels, light text, accent brightened `#2d5fa3`→`#4d8ef0` for contrast on dark). `color-scheme: light dark` on `:root` plus `<meta name="color-scheme" content="light dark">` in `index.html` theme native form controls and scrollbars. The last literal hex tints that would have stayed light (`.btn-danger:hover`, Dashboard `.denied-box`, the badge backgrounds) became new `--*-subtle` tokens. Charts theme automatically since they use tokens. OS-follow only — an explicit toggle is a possible follow-up. Built on `issue/0051` (Sonnet implement, Opus review), squash-merged to `main` as `4c94518`.

## Verification

From `web/`: `npm run check` → 0 errors; `npm run build` → clean; `npm test` → 162 passing (CSS/markup only, no logic touched; light `:root` values byte-identical to before). Independently-computed WCAG contrast (both themes): body text 14–16:1, muted text 4.95–6.15:1, accent 5.2–7.3:1 — all ≥ AA. Reviewer caught one AA failure — dark `--text-faint` `#6e7681` (3.77:1) on the 10px `ClicksChart` axis labels — which was fixed by raising it to `#8b949e` (panel 5.62:1, page 6.15:1, subtle 5.26:1) and re-verified.

## Files changed

- `web/src/app.css` — `@media (prefers-color-scheme: dark)` token override, `color-scheme`, new `--success-subtle`/`--danger-subtle`/`--danger-hover-bg` tokens.
- `web/index.html` — `<meta name="color-scheme" content="light dark">`.
- `web/src/lib/Button.svelte` — `.btn-danger:hover` hex → `var(--danger-hover-bg)`.
- `web/src/views/Dashboard.svelte` — `.denied-box` hex → `var(--danger-subtle)`.

## Gotchas

- Token-only approach: dark mode is entirely value-restatement; no component CSS changed — this is the payoff of the #0052 token foundation.
- **Dark rendering not visually confirmed** (no headless browser). Eyeball with OS set to Dark: panel separation, accent vibrancy, the desaturated status tints (`--danger-subtle #2d1212` behind `--danger #f85149`), and chart area-fill legibility.
- Pre-existing (out of scope, flagged by reviewer): light-mode `--text-faint` (`#8a8d91` ≈ 2.8:1 on white) has the same small-text AA concern in *light* mode — predates this branch; worth a follow-up.
- Follow-up option: explicit light/dark/system toggle with `localStorage` (one header button + a few lines).
