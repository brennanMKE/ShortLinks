## Description

Implement UTM analytics: aggregate click counts broken down by utm_source, utm_medium, and utm_campaign. Surfaced via the link detail API endpoint.

## Acceptance Criteria

- [x] `GET /api/links/{key}` response includes a `utm_stats` field with aggregated counts:
  ```json
  {
    "utm_stats": {
      "by_source": [{"value": "email", "count": 42}, ...],
      "by_medium": [{"value": "newsletter", "count": 30}, ...],
      "by_campaign": [{"value": "launch", "count": 42}, ...]
    }
  }
  ```
- [x] NULL utm values are grouped as `"(none)"` in the aggregation
- [x] Results ordered by count descending, limited to top 20 per dimension
- [x] Total click count included in response: `"click_count": <int>`
- [x] Queries run against the `clicks` table (no separate aggregate table needed for Phase 4)
- [x] `go test ./internal/handlers/` covers: link with UTM clicks returns correct breakdown, link with no clicks returns empty arrays

## Implementation

To make analytics real, this issue also closed the click-recording loop that
earlier issues left as interface seams.

- **Click recorder** (`internal/clicks/recorder.go`): `Recorder` satisfies the
  redirect handler's `ClickRecorder`. It resolves the short key to `link_id`
  inside the INSERT (a scalar subquery over `links`), so an unknown/deleted key
  records zero rows rather than erroring. Empty-string metadata and unparseable
  IPs fold to SQL `NULL` so the analytics `(none)` bucket reflects genuine
  absence. `RecordClick` is fire-and-forget (bounded background context,
  log-and-continue) so it never blocks or breaks a redirect; `Record` returns an
  error for tests.
- **UTM aggregation** (`internal/clicks/stats.go`): `StatsStore.UTMStatsForLink`
  returns total `click_count` plus `by_source`/`by_medium`/`by_campaign`
  breakdowns. `COALESCE(NULLIF(col,''), '(none)')` folds NULL and empty into one
  `(none)` bucket; each dimension is ordered count desc, value asc, limited to
  top 20. Empty slices are non-nil so JSON encodes `[]` not `null`. Dimension
  column names are validated against an allow-list before interpolation.
- **Real link resolver** (`internal/links/resolver.go`): `Resolver` implements
  the handler's `LinkResolver` — checks the redirect `cache.Cache` first, falls
  back to the new `Store.ResolveByKey` (non-user-scoped, since `/u/{key}` is
  public), caches positive hits and short-TTL negative entries on a DB miss, and
  converts a negative hit to `found=false` for the handler.
- **Redirect wiring** (`cmd/shortlinks/main.go`): `serve()` now constructs the
  redirect cache, resolver, clicks recorder, and stats store; mounts
  `GET /u/{key}`; and passes the redirect cache to `LinksHandler` as the
  cache-evictor (replacing the previous `nil`) so PATCH/DELETE invalidate it.
- **Analytics surface** (`internal/handlers/links.go`): `GET /api/links/{key}`
  now returns `linkDetailView` = the existing link fields plus a `utm_stats`
  object (`click_count`, `by_source`, `by_medium`, `by_campaign`). The link is
  resolved scoped to the authenticated user first, then the breakdown query runs
  against the resolved `link.ID`, so a non-owner gets 404 and never sees another
  user's breakdown. `NewClickRecorder` (in `redirect.go`) adapts the handler's
  `ClickInfo` to `clicks.Click` so the clicks package never imports handlers.

## Gotchas

- The redirect path was an unfinished seam: `internal/handlers/redirect.go`
  already defined `LinkResolver`/`ClickRecorder`, but `internal/clicks` was a
  `doc.go` stub, no resolver existed, `GET /u/{key}` was NOT mounted in
  `serve()`, and `LinksHandler` got a `nil` cache-evictor. This issue closed all
  of that — redirects + click recording now work end-to-end.
- `NewLinksHandler` gained a sixth parameter (`statsProvider`); all existing call
  sites (4 tests + main) were updated.
- `clicks.NewRecorder` resolves key→link_id in SQL rather than taking a link id,
  matching the handler's `ClickInfo.Key` (the handler never has the id).
