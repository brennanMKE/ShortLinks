## Description

On the Dashboard "Create a short link" form, the **Expires (optional)** input is wider than the other text inputs (Destination URL, Title (optional), Custom alias (optional)). It runs to the panel's right edge while the others stop short, so the fields no longer line up. Expires should be the same width as the other text fields.

## Steps to reproduce

1. Sign in at https://go.sstools.co/ and stay on the Dashboard.
2. In the "Create a short link" panel, compare the **Expires (optional)** field to Destination URL / Title / Custom alias.
3. Observe the Expires box is wider (flush to the panel edge) than the others.

## Expected behavior

The Expires input is the same width as the other text inputs in the form.

## Actual behavior

The Expires input is wider than the other inputs, breaking the form's alignment.

## Notes

Likely the Expires control (a `datetime-local` input) isn't constrained to the same width / `box-sizing` as the other fields. Shared input styling lives in `web/src/app.css` (global form-control rules); the create form is in `web/src/views/Dashboard.svelte`. Check whether the Expires input sits in the same `.field` wrapper and gets `width: 100%` like the others, or whether the native datetime input's intrinsic sizing is overriding it. Surfaced on mobile Safari in dark mode, but the discrepancy is layout, not theme-specific.

## Root cause

The Expires field is a native `datetime-local` input. On iOS Safari that control has a **wide intrinsic `min-width`** (date + time + AM/PM chrome) which beats `width: 100%` — `width: 100%` does not override the intrinsic `min-width: auto` of a native control — so it stretched past its `.field`/panel and rendered wider than the text inputs. The input was already in the shared `app.css` form-control rule with `width: 100%`, and `box-sizing: border-box` already applied via the universal `*` reset, so neither of those was the missing piece. (The note's "wrapper mismatch" guess was wrong — the Expires field uses the same `.field` markup as its siblings.)

## Fix

Added `min-width: 0` scoped to `input[type="date"]`, `input[type="time"]`, and `input[type="datetime-local"]` in `web/src/app.css`. `min-width: 0` defeats the native control's intrinsic minimum width, letting `width: 100%` win so the field is constrained to its container and matches the other inputs. An earlier attempt added `max-width: 100%` + `box-sizing: border-box`, but the reviewer correctly flagged those as a no-op/redundant (a `width:100%` border-box element can't exceed its container); they were dropped in favor of `min-width: 0`.

## Gotchas

- `width: 100%` does **not** override a native control's intrinsic `min-width` — `min-width: 0` is the load-bearing property for `datetime-local`/`date`/`time` on iOS. `max-width: 100%` looks plausible but is a no-op when `box-sizing` is already `border-box`.
- This is iOS-Safari-specific intrinsic-sizing behavior; desktop browsers already rendered the field at container width, so there's no desktop regression and no desktop way to reproduce the original symptom.

## Follow-up — first fix (min-width:0) was insufficient on iOS (2026-06-22)

After deploying, the Expires field was **still wider** than the text inputs on iPhone. Confirmed from the live bundle that the `min-width:0` rule had genuinely shipped (`curl` of the served CSS showed `input[type=datetime-local]{min-width:0}` and `width:100%`) — so this was **not** a deploy/cache problem; the CSS was simply insufficient.

Reclassified root cause: iOS Safari renders date/time inputs as **native controls that ignore `width:100%` entirely** until the native appearance is removed — `min-width:0` does not override that. Added `-webkit-appearance: none; appearance: none` (kept `min-width:0`) in `web/src/app.css` (commit `955edf1`); removing the native appearance makes the control a normal styleable box that obeys `width:100%`, matching the other fields.

**Status is `in-progress`, not `resolved`** — the previous `resolved` was premature (the fix was reasoned but unverifiable headlessly and turned out not to work). This iOS-specific rendering **cannot be reproduced in a headless/desktop environment**, so it needs an on-device check. Fastest certain verification: connect the iPhone to the Mac, Safari → **Develop → [iPhone] → go.sstools.co**, inspect the Expires `<input>`, and confirm `-webkit-appearance: none` collapses it to the sibling width before trusting another deploy.
