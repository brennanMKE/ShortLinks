## Description

Implement the Account view (`web/src/views/Account.svelte`) for managing registered passkeys: viewing all credentials, renaming them, and revoking them (with a guard against revoking the last one).

## Acceptance Criteria

- [ ] Loaded when `currentView === "account"`
- [ ] On mount, calls `GET /account/credentials` to list passkeys
- [ ] Each credential row shows: `device_name`, `created_at`, `last_used_at`; has inline rename (editable field + save) and revoke (button + confirmation) actions
- [ ] Rename: calls `PATCH /account/credentials/{id}` with `{device_name}`; updates list in place on success
- [ ] Revoke: calls `DELETE /account/credentials/{id}`; if server returns `409` (last credential), shows error "You cannot revoke your only passkey. Add another passkey first."
- [ ] On successful revoke, removes credential from list
- [ ] Component is shown when `currentView === "account"`

## Notes

Dependencies: #0019 (credential management API), #0031 (stores, navigation).

## Implementation

Replaced the `web/src/views/Account.svelte` placeholder with a full passkey
management view, matching the Svelte 5 runes + error-handling style of
`LinkDetail.svelte`/`Dashboard.svelte` and reusing the topbar/tabs/sign-out nav
for consistency with the other views.

- **List**: on mount calls `listCredentials()` (`GET /account/credentials`) with
  loading, error (+ Retry), and empty states. Each row renders the device name
  (inline editable), `device_hint` (computed server-side from the AAGUID),
  created + last-used dates, and the sign count.
- **Rename**: inline edit form calling `renameCredential()`
  (`PATCH /account/credentials/{id}` with `{device_name}`); validates non-empty,
  updates the row in place with the authoritative server record on success, and
  shows per-row errors (404 → "This passkey no longer exists.").
- **Revoke**: confirmation prompt then `revokeCredential()`
  (`DELETE /account/credentials/{id}`); removes the row on success. The
  **409 `cannot_revoke_last_credential`** case shows the actionable message
  "You cannot revoke your only passkey. Add another passkey first." rather than a
  generic error. The Revoke control is also disabled when only one credential
  remains (PRD guard), with a tooltip explaining why.
- Shows the current user's email (from `currentUser`) and Dashboard/Admin tabs.

**No "add passkey" affordance** was included: the issue AC does not require one,
and per the PRD a logged-in user adds a credential through the account-recovery
flow — there is no logged-in add-credential endpoint in the routes.

Pure logic was factored into `web/src/lib/account.ts` and unit-tested in
`web/src/lib/account.test.ts`: date/last-used formatting, rename validation, the
`canRevoke` last-credential guard, and the revoke-failure → friendly-message
mapping (matched by 409 status AND the `cannot_revoke_last_credential` code).

The Go backend was not modified; the existing `internal/handlers/credentials.go`
contract (fields, rename body, 409 refusal) was matched exactly.

## Gotchas

- The backend already returns a `device_hint` (AAGUID→label mapped server-side
  in `internal/handlers/credentials.go`), so no client-side AAGUID mapping was
  needed — the view just renders the provided hint.
- The 409 mapping keys off both the HTTP status and the
  `cannot_revoke_last_credential` error code so a future status change still
  resolves to the friendly copy.
