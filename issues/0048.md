## Description

When creating or editing a short link, give the user a dedicated UTM builder — labelled fields for `utm_source`, `utm_medium`, `utm_campaign`, `utm_term`, `utm_content` — that compose onto the destination URL, rather than requiring the user to hand-type query strings. The redirect already passes UTM params through and the click stats already break down by UTM dimension (`internal/clicks/stats.go`, surfaced in `LinkDetail.svelte`); this closes the loop on the *authoring* side so the captured dimensions are intentional and consistent.

## Motivation

UTM tracking exists end-to-end on the read path, but on the create path the destination is a single free-text URL field (`web/src/views/Dashboard.svelte`). Users must manually append `?utm_source=…&utm_medium=…`, which is error-prone (typos, inconsistent casing, duplicate keys, encoding mistakes) and produces messy analytics that undercut the existing per-UTM stats view.

## Proposed approach

- Add a collapsible "Campaign / UTM parameters" section to the link create (and edit, if supported) form with the five standard fields.
- Compose them onto the destination URL safely: merge with any existing query string, URL-encode values, drop empty fields, avoid duplicate keys.
- Show a live preview of the final destination URL as fields change.
- Keep the composition logic as pure functions in `web/src/lib/` (mirroring the existing `lib/linkDetail.ts` pattern) so it is unit-testable without the DOM.
- Decide and document whether UTM values are stored as discrete fields or only baked into the destination URL (affects whether edit can re-populate the builder).

## Implementation

Added a collapsible "Campaign / UTM parameters" section to the Dashboard link-create form (five standard fields + a live composed-destination preview), built on the #0052 design system (`Panel`/`Button`/tokens, global form styles — no hardcoded hex except one decorative 9px chevron). Composition lives in pure DOM-free helpers `web/src/lib/utm.ts` (`composeUtmUrl`, `isUtmEmpty`, `emptyUtmParams`, `UTM_KEYS`): merges UTM params onto any existing query via `URLSearchParams` (correct encoding, duplicate-key replacement, empty-field dropping, fragment + port preserved), with a string fallback for transient invalid/relative previews. **Storage decision:** UTM values are baked into `destination_url` at submit — no backend/API/DB change — so the redirect passthrough and per-UTM `LinkDetail` stats work unchanged. Built on `issue/0048` (Sonnet implement, Opus review → APPROVE), squash-merged to `main` as `e159328`.

## Gotchas

- Svelte 5: `bind:value={utmParams.utm_source}` mutates a `$state` object property — reactive, drives the `$derived` preview correctly.
- Edit re-population not supported: baking into the URL means a future edit form can't pre-fill the five fields without parsing the stored URL (lossy) or adding discrete columns — flagged for a follow-up if edit-UTM is wanted.
- Known cosmetic follow-up (reviewer, non-blocking): `appendQueryFallback` doesn't split on `#`, so the *live preview* of a relative URL with a fragment renders malformed. Unreachable for persisted links — `isValidHttpUrl` (Dashboard) + server `validDestinationURL` gate submission to the correct `new URL()` path. Fix: split fragment in the fallback + add a test.
