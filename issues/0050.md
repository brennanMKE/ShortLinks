## Description

Adapt the management SPA to iPhone-class viewport widths (~375–430px) so links can be created, viewed, and managed comfortably on a phone. The `<meta name="viewport">` is already present (`web/index.html:5`), but layouts use fixed `max-width` cards and the list/table views (Dashboard link list, Admin, LinkDetail) are not audited for narrow screens.

## Motivation

The app is reached at `go.sstools.co` from any device; short-link management is a natural thing to do from a phone. Today's fixed-width cards and wide tables overflow or get cramped on mobile.

## Proposed approach

- Audit each view (`Login`, `Dashboard`, `LinkDetail`, `Account`, `Admin`, `RegisterVerify`, `RecoverVerify`) at iPhone widths in browser device emulation.
- Replace fixed pixel/rem widths with fluid constraints (`max-width` + `width: 100%`, sensible padding); add breakpoints where layout must reflow.
- Make tabular/list data (link lists, audit log, per-UTM stats) reflow or scroll deliberately rather than overflow the viewport.
- Ensure tap targets (buttons, links) meet a usable minimum size; check the passkey ceremony flows render on mobile Safari.
- Verify any charts from [#0049](0049.md) scale down without breaking.

## Acceptance criteria

- [x] All views usable and non-overflowing at 375px and 430px widths
- [x] Link create/list, LinkDetail, Account, Admin reflow cleanly; no horizontal scroll except where intentional
- [x] Tap targets are comfortably sized on mobile

## Implementation

Made the SPA usable at iPhone widths (375–430px). Added a shared `@media (max-width:480px)` block to `web/src/app.css` (`.app-shell`/`.app-header` gutters + `flex-wrap`) and a `.table-scroll` utility. Wide tables (Dashboard link list; Admin users / audit log / filter rules) are wrapped in `.table-scroll` inside their `Panel` so they scroll horizontally within a bounded container rather than overflowing the viewport. Nav-tabs reflow onto their own row on phones (`order`/`flex:0 0 100%` on a genuine flex child of the `flex-wrap` header), the LinkDetail field grid collapses to one column, Account credential cards stack, and buttons/tabs get ≥40px tap targets. CSS/markup only — no behavior change; every rule is additive behind a media query so desktop is untouched. Built on `issue/0050` (Sonnet implement, Opus review → APPROVE), squash-merged to `main` as `e690e82`.

## Verification

From `web/`: `npm run check` → 0 errors; `npm run build` → clean; `npm test` → 162 passing (no logic/test files touched, so counts unchanged by construction). The reviewer resolved the two flagged "medium-risk" items **from the CSS cascade with file:line evidence**: (1) the nav-tabs reflow is structurally valid — `.nav-tabs` is a direct flex child of `.app-header { display:flex; flex-wrap:wrap }` in Dashboard/Account/Admin; (2) every wide table is double-bounded (`.table-scroll` `overflow-x:auto` + `Panel` `overflow:hidden` inside the constrained `.app-shell`), so horizontal viewport overflow is impossible.

## Gotchas

- **No headless browser was available** (no Chrome/Chromium; Playwright CLI present but no browser binaries), so pixel-level layout at exactly 375/430px was NOT visually confirmed — correctness was established from the cascade instead. What still warrants a human eyeball is purely **aesthetic**, not layout-breaking: balance of the wrapped nav-tabs row, vertical rhythm of the 40px tap targets in Account cards, and discoverability of the horizontally-scrolling Admin tables (no visible scroll affordance). Open `cd web && npm run dev` + Chrome DevTools at iPhone SE (375) / 14 Pro Max (430) to confirm.
- Charts (#0049) were already `viewBox`+`width:100%` and needed no change to scale down.
