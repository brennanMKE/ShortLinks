## Description

Introduce a storage abstraction so the app can run against a **JSON-file** (or in-memory) backend in development instead of PostgreSQL. Today the stores talk to `pgxpool` directly (`internal/links`, `internal/clicks`, `internal/auth`, sessions, settings, users, audit, filters). Define interfaces over the data each store needs and provide a dev implementation backed by JSON files on disk, selected via config.

## Motivation

Postgres + `golang-migrate` is required just to boot the app, which blocks running it locally for quick UI checks. A file-backed dev store removes that dependency for development. (Part of [#0056](0056.md).)

## Proposed approach

- Define Go interfaces for the persistence each handler needs (links CRUD + lookup, users, sessions, settings, clicks/stats, audit, filter rules). The existing Postgres implementations satisfy them unchanged.
- Add a JSON-file-backed implementation (per-entity files under a dev data dir, or a single JSON document, serialized on write). An in-memory variant is fine if simpler — but the request is JSON files, so default to file-backed with persistence.
- Select the backend via config: when `DATABASE_URL` is empty or `STORAGE=json`, wire the JSON store; otherwise Postgres. Default/production stays Postgres.
- Scope to the stores the dev flows exercise (links create/list/detail, the mock user/session from #0058, settings); clicks/stats may return empty in dev. Note any store left as a stub.
- Guard file writes with a mutex — this is a single-process dev convenience, not a production store.

## Acceptance criteria

- [x] Storage interfaces defined; Postgres implementations satisfy them with no behavior change
- [x] A JSON-file dev backend implements the interfaces for the dev-critical stores
- [x] Backend selected by config; production path defaults to Postgres and is unchanged
- [x] The app boots and serves the dashboard against the JSON backend with no Postgres running
- [x] `go build ./...` and `go test ./...` pass (Postgres-backed tests still run against the test DB)

## Implementation

The HTTP handlers already depend on consumer-side interfaces (`linkStore`, `statsProvider`, `ruleProvider`, `auditReader`, `credentialStore`, `LinkResolver`, `Pinger`, etc.), so no handler refactor was needed. Added `internal/devstore` — a single in-memory `*Store` (mutex-guarded maps) that satisfies all of those interfaces with compile-time `var _ = ` assertions. `cmd/shortlinks/main.go` now branches on `config.DevMode()`: `serveDevMode` constructs the dev store and skips the pgxpool connect entirely; `servePostgres` is the original `serve()` body extracted verbatim. Selection is by an **explicit `STORAGE=json`** only — an empty `DATABASE_URL` does NOT engage dev mode, so production can never silently fall back to the in-memory store. The dev store is seeded with a mock admin (`ADMIN_EMAIL` or `admin@localhost`) and sample links (`wiki`, `gh`) so the UI isn't empty. **In-memory** was chosen over JSON files for reliability (the goal is local UI validation, not durable data). Built on `issue/0057` (Sonnet implement, Opus review → APPROVE), squash-merged to `main` as `830c712`.

## Verification

`go build ./...` clean; `go test ./...` (with the local `shortlinks_test` DB) — existing tests pass (Postgres path unchanged), 8 new `internal/devstore` tests pass, `-race` clean. (The only failure, `internal/config TestLoad_DefaultsApplied`, fails identically on `main` — pre-existing, unrelated.) **Boot-without-Postgres proof (re-run by the reviewer):** `STORAGE=json ./shortlinks serve` with no `DATABASE_URL` starts cleanly; `/health` → `200 {"status":"ok"}`, `/` → the SPA `index.html`, `/u/wiki` → `302 https://www.wikipedia.org`, unknown key → `404`.

## Gotchas

- Dev mode is `STORAGE=json` ONLY — never an empty `DATABASE_URL`. Comments were corrected to say so; do not loosen this (it would let prod silently lose data to the in-memory store).
- Clicks/audit are empty no-ops in dev and passkey ceremonies are stubs; the authenticated dashboard still needs a session until #0058 adds the dev login bypass. The public paths (`/`, `/health`, `/u/{key}`) fully work.
