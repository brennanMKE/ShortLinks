## Description

Implement the server-side reading and enforcement of the `registrations_enabled` setting. The setting is read from the DB on each registration attempt (not cached) so admin changes take effect immediately without a restart.

## Acceptance Criteria

- [x] `POST /auth/register/start` reads `registrations_enabled` from `settings` table on every call; returns `403` when value is `"false"` (already implemented in #0015 via `Store.RegistrationsEnabled`; the handler returns `403 {"error":"Registration closed"}`)
- [x] `GET /admin/settings`: returns all rows from the `settings` table as `{"settings": [{"key": ..., "value": ...}]}`; requires admin auth
- [x] `PATCH /admin/settings`: accepts `{"key": "registrations_enabled", "value": "true"}`; updates `settings` table; returns `200`. (The `settings.updated` audit entry is #0025 — a TODO/no-op here; the old value is already captured in `Store.UpdateSetting` so #0025 can emit `{key, old_value, new_value}` verbatim.)
- [x] Only keys that already exist in the `settings` table can be updated (no arbitrary key creation via this endpoint); unknown keys return `400`
- [x] `go test ./internal/handlers/` covers: admin settings read/write, invalid value rejected, unknown key rejected, non-admin blocked (403) and unauthenticated (401) on both routes, and an end-to-end gate tie-in (PATCH→`/auth/register/start` no longer 403)

## Notes

Dependencies: #0013 (settings table seeded with `registrations_enabled = false`), #0015 (register/start calls this check), #0017 (admin auth middleware).

## Implementation

Added the admin-only settings READ/WRITE endpoints; the registration enforcement
already existed (see Gotchas).

- `internal/auth/store.go` — extended the pgx `Store` with settings queries:
  - `ListSettings(ctx)` → `[]auth.Setting{Key, Value, UpdatedAt}` ordered by key (backs GET).
  - `GetSetting(ctx, key)` → value, `ErrSettingNotFound` when absent.
  - `UpdateSetting(ctx, key, value, now)` → updates an **existing** key and bumps
    `updated_at`, returning the previous value (for the future #0025 audit entry).
    Uses a `WITH prev AS (...) UPDATE ... FROM prev RETURNING prev.old_value` so the
    old value and the missing-key detection (`ErrSettingNotFound`) come from one
    atomic statement. No `INSERT`/`ON CONFLICT` — the endpoint must not create
    arbitrary keys, so a missing key is an error, not an upsert.
- `internal/handlers/settings.go` — new `SettingsHandler` (depends on a
  `settingStore` interface, mirroring the existing handler pattern):
  - `List` → `GET /admin/settings`, returns `{"settings":[{key,value,updated_at}]}`
    (slice always non-nil; `updated_at` omitted when NULL).
  - `Patch` → `PATCH /admin/settings`, body `{"key","value"}`. Validates the value
    for known keys (`registrations_enabled` must be `"true"`/`"false"`), maps
    `ErrSettingNotFound` → `400 unknown setting key`, invalid value → `400`, then
    returns the full updated settings list. `settings.updated` audit is a documented
    #0025 TODO/no-op (old value captured, ready to emit).
- `cmd/shortlinks/main.go` — constructed `settingsH`, added a `requireAdmin`
  composer (`RequireSession` then `RequireAdmin`), and wired
  `GET /admin/settings` and `PATCH /admin/settings` behind it.

## Gotchas

- The registration **enforcement** (`POST /auth/register/start` reading
  `registrations_enabled` fresh per attempt and returning 403 when false) was
  already implemented in #0015 via `RegistrationService.StartRegistration` →
  `Store.RegistrationsEnabled`. This issue only added the admin READ/WRITE
  endpoints; the tie-in test confirms the existing gate honors a live toggle.
- The handler returns `403 {"error":"Registration closed"}` (not the literal
  `registration_closed` the criteria's example showed) — that string predates this
  issue (#0014/#0015) and was left unchanged to avoid touching the registration
  handler's contract.
- The `settings` table is not in the credentials suite's `TRUNCATE` list, so the
  settings test pool resets that row to the seed (`false`) in setup and teardown to
  stay isolated from the auth suite (which flips it) and to leave the DB clean.
