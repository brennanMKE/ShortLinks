# 0046 — Magic-link landing URL is never cleared after the ceremony, so a reload replays a consumed token and dead-ends

| | |
|---|---|
| **Status** | open |
| **Module** | web |
| **Platform** | All |
| **First seen** | 2026-06-17 |

## Description

After completing the account-recovery ceremony, the SPA flips the in-memory view to the dashboard but leaves the address bar on `/recover/verify?token=…`. Recovery tokens are single-use with a 15-minute TTL, so reloading the page re-enters the recovery flow against a now-consumed/expired token, fails with 410/400, and dead-ends on the "This recovery link has expired or is invalid" error screen — even though the user's session cookie is valid and they are actually signed in. After landing the new passkey the URL should be replaced with `/` so a reload lands in the normal authenticated state.

## Long Description

`App.svelte` routes the initial view entirely from `window.location.pathname` (`App.svelte:38-45`): when the path is `/recover/verify` it reads `?token=` into the `pendingVerifyToken` store and shows `RecoverVerify`. On success, `RecoverVerify.svelte:128-133` calls `GET /api/me`, sets `currentUser`, clears `pendingVerifyToken`, and sets `currentView = 'dashboard'` — but it never touches `window.location`. The URL still reads `/recover/verify?token=…`.

A reload re-runs `App.svelte`'s `onMount`, sees the recovery path again, and replays the same token through `recoverVerify()`. The server already consumed it (and/or its 15-min TTL lapsed), so it returns 410/400 and `RecoverVerify` renders the expired-link error. The user appears "logged out" / "session expired" despite holding a valid 30-day session cookie — the only escape is to manually edit the URL back to `/`.

The same defect applies to the registration landing `/register/verify` (`RegisterVerify.svelte`), which follows the identical pattern and token semantics. The fix should cover both magic-link landings.

## Steps to reproduce

1. Request an account-recovery email and click the link → land on `/recover/verify?token=…`.
2. Complete the passkey-creation prompt; the app shows the dashboard.
3. Reload the page (or close/reopen the tab on the same URL).
4. Observe the "This recovery link has expired or is invalid" error screen instead of the dashboard.

## Expected behavior

Once the recovery (or registration) ceremony succeeds, the URL is replaced with `/` (a history *replace*, not push, so Back doesn't return to the consumed-token URL). A subsequent reload routes through `GET /api/me`, finds the valid session, and lands on the dashboard.

## Actual behavior

The URL stays on `/recover/verify?token=…`. Reloading replays the consumed token, returns 410/400, and strands the user on the expired-link error screen despite a valid session.

## Suggested fix

- In `RecoverVerify.svelte`, on the success path (step 4, after `getMe()` resolves) call `history.replaceState({}, '', '/')` (or the project's router equivalent) before/with setting `currentView = 'dashboard'`.
- Apply the same to `RegisterVerify.svelte`'s success path.
- Consider also stripping the `?token=` from the URL defensively as soon as the token is captured in `App.svelte` (after reading it into `pendingVerifyToken`), so even a mid-ceremony reload can't replay it — while still preserving the in-memory token for the finish POST.
- Verify Back-button behavior: a `replaceState` should leave no consumed-token entry in history.
